Section 1 · Agent patterns & types

Agent patterns & agent types

Questions like "which agent type fits this task?" are really "which shape of the perceive→reason→act→observe loop fits?" NAT ships eight agent types — learn what each one's loop looks like and you can derive the answer.

🧠

Core loop & patterns

5 terms
Agentvs pipeline/workflowD1D2

A system that perceives input, reasons, acts via tools, and observes results — changing its behavior based on intermediate observations (a fixed pipeline can't).

NVIDIA: Authored in NeMo Agent Toolkit (NAT) via YAML, picking one agent-type implementation.

Exam: If the task can change course mid-run from a tool result, it wants an agent; if it's the same every time, a workflow.

Agent loopperceive → reason → act → observeD1D2

The four-phase cycle every agent repeats until a stop condition. Agent types differ in the shape of this loop.

NVIDIA: ReAct loops every step; ReWOO plans all steps first; Tool Calling runs ~one act→observe.

Exam: Memorize the loop and "which agent type?" becomes derivable.

Multi-agent systemD1D5

Multiple specialized agents that coordinate — via router/orchestrator, sequential handoff, hierarchy, or peer messaging — to solve a task no single agent owns.

NVIDIA: Composed in NAT; cross-process coordination uses A2A. Reference = Multi-Agent Warehouse Blueprint.

Exam: Know the coordination shapes (router / sequential / hierarchical / A2A peer) and when each applies.

Compound errorwhy agents are hardD1D3

End-to-end success ≈ p_step ^ n_steps — per-step accuracies multiply.

NVIDIA: Drives "fewer steps / higher per-step reliability / verification" guidance.

Exam: 8 steps × 92% → 0.92⁸ ≈ 0.51. The fix is fewer/hardened steps, not "a bigger model."

Chain-of-Thought (CoT)D5D2

Prompting the model to emit intermediate reasoning before the final answer, improving complex-reasoning accuracy.

NVIDIA: Used by NAT ReAct & Reasoning agents; test-time compute extends it.

Exam: CoT + strict schema → put the reasoning field first so the model reasons before committing.

🤖

The 8 NAT agent types

8 terms
ReActReasoning + ActingD1D2D5

Interleaves Thought → Action → Observation, re-planning after every observation.

NVIDIA: NAT agent type; routes to tools via their names/descriptions.

Exam: Best for adaptive multi-step tasks; highest token/latency cost. The default autonomous agent.

ReWOOReasoning Without ObservationD1D2D5

Generates the entire plan in one LLM call, then executes with no reasoning between steps.

NVIDIA: NAT agent type; decouples planning from execution for token efficiency.

Exam: Cheaper/faster than ReAct for predictable tasks, but can't adapt mid-run. The "ReAct vs ReWOO" trade = adaptivity vs cost.

Reasoning AgentD2D3D5

Reasons through planning ahead of time and invokes an underlying function/agent — unlike ReAct, it doesn't reason between every step. Vehicle for test-time compute.

NVIDIA: NAT agent type; wraps another agent/function.

Exam: Reach for it when hard-reasoning quality justifies extra inference. Don't confuse with ReAct (which reacts each step).

Tool Calling AgentD2

Makes a single tool selection + execution per turn — no multi-step loop.

NVIDIA: NAT agent type; lowest latency/cost.

Exam: Right for simple, well-defined tasks where the tool is obvious. "Minimize latency, single clear action" → this.

Router AgentD1D2

Classifies the input and dispatches to a specialist agent — it routes, it doesn't do the task.

NVIDIA: NAT agent type; typical entry point of a multi-agent system (intent / keyword / LLM classification).

Exam: "Billing → billing agent, tech → support agent" = Router. Distinguish from Sequential and Parallel.

Sequential ExecutorD1D2

Runs a fixed, ordered sequence; each step feeds the next; every step always runs (no routing).

NVIDIA: NAT agent type for deterministic pipelines (extract → transform → validate → store).

Exam: The "agent" answer when the process is the same every time — no branching.

Parallel Executorthe easily-forgotten oneD1D2

Runs multiple sub-agents/functions concurrently and aggregates results.

NVIDIA: NAT agent type — the one most often omitted from third-party "list the agent types" sheets.

Exam: Use when sub-tasks are independent and can run at once to cut latency. If you memorized "seven," you missed this — there are eight.

Responses API AgentD2

A NAT agent built around the OpenAI-style Responses API interaction shape (stateful, multi-turn, tool-augmented).

NVIDIA: One of NAT's eight agent types.

Exam: Appears in "name all NAT agent types" questions — know it's the Responses-API-shaped one.

Section 2 · The NVIDIA stack

NVIDIA platform — NAT, NeMo, NIM

The exam's center of gravity. NAT orchestrates agents; NeMo microservices supply customization, evaluation, data, retrieval, and safety; NIM serves the models. "Which NVIDIA product does X?" is a recurring shape — match the job to the exact name.

🧰

NeMo Agent Toolkit (NAT)

4 terms
NAT (NeMo Agent Toolkit)formerly AIQ / Agent Intelligence ToolkitD1–D8

Open-source toolkit to build, deploy, evaluate, profile & monitor agents. YAML config; ships agent types, deploy servers (FastAPI/MCP/A2A), eval framework, profiler, middleware, plugins.

NVIDIA: pip install nvidia-nat; CLI prefix nat. Framework-agnostic (LangChain, LlamaIndex, CrewAI). Base install = core agents/config/functions/middleware; eval/profiling extras may need extra packages.

Exam: Touches almost every domain. Know what's in the base vs the extras.

Function / Function GroupD2

A function is a registered tool; a Function Group bundles functions assignable to an agent as a unit.

NVIDIA: Part of NAT's function registry; makes tool sets modular/reusable.

Exam: "Give several agents the same toolset cleanly" → Function Group.

Middleware (NAT)D2D8D9

Pipeline components between caller and agent core handling cross-cutting concerns.

NVIDIA: NAT middleware for caching, logging, defense/security, red-teaming, configured in YAML.

Exam: "Add caching/logging/safety without touching agent logic" → middleware.

Plugin system (NAT)D2D3

NAT's extension mechanism — register custom agents, tools, evaluators, front-ends, integrations.

NVIDIA: Basis for the function/evaluator registries; extend NAT without forking.

Exam: "Add a custom evaluator / tool to NAT" → register a plugin.

🏭

NeMo microservices

5 terms
NeMo CustomizerD3D7

Fine-tuning microservice — SFT, LoRA, and DPO — via API jobs.

NVIDIA: The fine-tune engine in the Data Flywheel; NAT's harness drives it (OpenPipe ART for GRPO).

Exam: "Run a managed DPO/SFT job" → Customizer.

NeMo EvaluatorD3D7

Evaluation microservice — academic benchmarks (MMLU, GSM8K), LLM-as-a-judge (any NIM), retriever/RAG metrics — via eval targets/configs/jobs.

NVIDIA: Standalone microservice; NAT eval runs inside the workflow instead.

Exam: "Benchmark a NIM + judge chats + score a retriever via API in NeMo" → Evaluator.

NeMo CuratorD6D8

GPU-accelerated data preparation — fuzzy dedup, language/quality filtering, extraction, PII removal — at corpus scale (10–100× CPU).

NVIDIA: The ingestion/curation stage feeding embeddings, fine-tuning, the Data Flywheel.

Exam: "Prepare millions of docs for indexing/training" → Curator (data prep, not retrieval/eval).

NeMo RetrieverD6D7

RAG retrieval microservice family — embedding, indexing, search, reranking — served as NIMs.

NVIDIA: e.g. llama-nemotron-embed-1b-v2 (embed) and llama-nemotron-rerank-1b-v2 (rerank). version-dependent these supersede the older nv-embedqa/nv-rerankqa v2 NIMs (2026-05-18 deprecation) — check the current model card.

Exam: "NVIDIA service for embedding + reranking in RAG" → NeMo Retriever NIMs. Embedding ≠ reranking.

NeMo (family)don't conflate the fiveD7

The five lifecycle microservices: Customizer (fine-tune) · Evaluator (eval) · Curator (data) · Retriever (RAG) · Guardrails (safety).

NVIDIA: Distinct products — "NeMo" the platform ≠ any one of them.

Exam: "Which NeMo microservice does X?" — memorize the five + one-line jobs.

📦

NIM & Blueprints

5 terms
NIM (NVIDIA Inference Microservice)D4D7

Standardized container serving a model with optimized inference, an OpenAI-compatible API, and health endpoints.

NVIDIA: LLMs, embedders, rerankers, and safety models all ship as NIMs. Hosted on build.nvidia.com or self-hosted.

Exam: "Self-hosted, standardized, OpenAI-compatible model endpoint" → NIM.

NIM OperatorD4

A Kubernetes operator automating NIM lifecycle — GPU allocation, model caching, health, inference-aware autoscaling.

NVIDIA: Recommended way to run NIM on K8s in production.

Exam: "Deploy/scale NIMs on K8s without hand-managing GPUs" → NIM Operator (vs Dynamo, the inference framework).

BlueprintD1D7

A complete deployable reference architecture composing NVIDIA tools for a use case (build.nvidia.com/blueprints).

NVIDIA: AI-Q · Multi-Agent Warehouse · Data Flywheel · Safety for Agentic AI.

Exam: Match the end-to-end use case to the named Blueprint.

AI-Q BlueprintD6D7

NVIDIA's RAG research-assistant Blueprint — end-to-end reference RAG (ingest → embed → Milvus+cuVS → rerank → NIM LLM).

NVIDIA: The canonical "production RAG wired with NVIDIA parts" example.

Exam: Full RAG stack + "the NVIDIA reference" → AI-Q.

Data FlywheelBlueprintD3D7D8

Continuous improvement: collect → evaluate → curate → fine-tune (smaller model) → redeploy after human review.

NVIDIA: Wires NAT (collect, tagged by workload_id) + Evaluator + Curator + Customizer.

Exam: "Cut inference cost using logged traffic without losing accuracy" → Data Flywheel.

Section 3 · Knowledge & retrieval

RAG — retrieval-augmented generation

RAG fixes knowledge/freshness/private-data gaps — fine-tuning fixes skills/behavior. The "RAG vs fine-tune" call is one of the most-tested distinctions. Evaluate retrieval and generation separately so the failing half tells you what to fix.

📚

Pipeline pieces

6 terms
RAGRetrieval-Augmented GenerationD6D3

Grounds generation in retrieved knowledge: search a KB for relevant chunks, pass them as context to the LLM.

NVIDIA: Curator → NIM embeddings → Milvus+cuVS → NIM reranker → NIM LLM; AI-Q is the reference.

Exam: RAG = facts; fine-tuning = skills. The core distinction.

EmbeddingD6

A dense vector where semantic similarity = geometric proximity, enabling similarity search.

NVIDIA: NeMo Retriever embedding NIMs (e.g. llama-nemotron-embed-1b-v2).

Exam: Embedding quality bounds recall — first tuning lever after chunking.

RerankingD6D3

A second-stage cross-encoder that re-scores first-stage candidates, catching query–doc interactions embeddings miss.

NVIDIA: NeMo Retriever rerank NIM (llama-nemotron-rerank-1b-v2, cross-encoder logit scorer; supersedes llama-3.2-nv-rerankqa-1b-v2); sits between retrieval and generation in AI-Q.

Exam: "Right docs retrieved but ranked too low" → add/tune a reranker (improves precision/NDCG).

ChunkingD6D3

Splitting documents into retrievable units (fixed / semantic / structure-aware), each embedded and indexed.

NVIDIA: The first RAG tuning lever; NVIDIA publishes chunking-strategy studies.

Exam: "Low recall@k, answers miss facts" → revisit chunking first.

Milvus / Vector DBD6

A database to store/index/search dense embeddings at scale (HNSW, IVF, flat; metadata filtering). Milvus is NVIDIA's standard.

NVIDIA: Milvus is the AI-Q vector store; integrates cuVS for GPU search.

Exam: NVIDIA-recommended production vector DB; index type trades recall vs speed/memory.

cuVSCUDA Vector SearchD6

GPU-accelerated ANN search (HNSW/IVF on GPU) for high-throughput concurrent vector search.

NVIDIA: Integrated into Milvus.

Exam: "Vector search is the bottleneck at high QPS" → cuVS/GPU search.

🎯

RAG generation metrics (RAGAS triad)

2 terms
Faithfulness / GroundednessD3D6

Are the answer's claims supported by the retrieved context? (claims supported ÷ total, 0–1).

NVIDIA: Scored via NAT / NeMo Evaluator LLM-judge; often gated ≥ ~0.8–0.9.

Exam: Low faithfulness + good context = a generation problem (hallucination), fix grounding/prompt — not retrieval.

Context relevance & Answer relevanceD3D6

The other two triad edges. Context relevance: is retrieved context relevant (low noise)? Answer relevance: does the answer address the question?

NVIDIA: RAGAS triad supported via NAT ragas / NeMo Evaluator.

Exam: Faithful-but-wrong → retrieval; relevant-context-but-unfaithful → generation. The RAG diagnosis pattern.

Section 4 · Measuring quality

Evaluation & metrics

Model benchmarks measure static knowledge; agent eval measures dynamic behavior over trajectories. Localize faults: a green light at one level doesn't imply the others pass.

📏

Eval methods & metrics

9 terms
Offline vs Online evaluationD3D8

Offline = pre-deploy, in CI, on golden/benchmark sets (repeatable, known failures). Online = live traffic (A/B, canary, feedback, drift — novel failures).

NVIDIA: Offline via NeMo Evaluator / NAT gates; online via observability + Data Flywheel feedback.

Exam: Complementary, not alternatives. "Catch novel real-world failures" → online; "block a bad change" → offline.

Golden datasetD3

A curated, version-controlled set of input → expected-output/trajectory pairs covering edge cases & past failures.

NVIDIA: Substrate for offline eval/CI gates; "every production bug becomes a test case."

Exam: Scores are meaningless without a versioned golden set pinned to the prompt/config. Quality > quantity.

LLM-as-a-JudgeD3

A strong LLM scores outputs against a rubric — pointwise, pairwise (A vs B), or reference-based.

NVIDIA: NAT eval & NeMo Evaluator with any NIM as judge.

Exam: Scalable proxy for humans — but mind the three biases. Prefer pairwise + position-swapping.

Judge biasesposition · verbosity · self-preferenceD3

Position: favors a fixed slot (often A). Verbosity: prefers longer answers. Self-preference: favors its own model family.

NVIDIA: Mitigate via position-swapping, conciseness rubrics, a different judge / jury.

Exam: "Flipping order flips the winner" → position bias; fix = swap orderings. Near-guaranteed content.

Trajectory evaluationD3D2

Scoring the sequence of actions (tools, args, order, recovery), not just the final answer.

NVIDIA: NAT eval — exact / in-order / any-order match vs a reference trajectory.

Exam: "Right answer via the wrong tools" → trajectory eval.

Task completion rate (TSR)D3

% of tasks where the agent achieved the goal end-to-end (task = intent + constraints). The headline metric.

NVIDIA: Common gate ≥ 85%; measure under normal, degraded & ambiguous scenarios.

Exam: Many gate questions reference an ≥85% TSR threshold.

Tool-call accuracyD3D2

Right tool, schema-valid correct arguments? Includes detecting hallucinated tools & overuse.

NVIDIA: Component of NAT trajectory/agent eval (target often >90%).

Exam: "Plan was right but the tool args were wrong" — a distinct failure layer.

Retrieval metricsRecall@k · Precision@k · MRR · NDCG@kD3D6

Recall@k = coverage of relevant docs in top-k; Precision@k = noise in top-k; MRR = how early the first relevant appears; NDCG@k = quality of the whole graded ranking.

NVIDIA: Computed by NeMo Evaluator retriever eval / NAT.

Exam: Low recall → fix chunking/embeddings/k; recall fine but bad answers → fix generation. MRR/NDCG matter (LLMs weight earlier context).

pass@k vs pass^kD3

pass@k = at least one success in k tries (capability). pass^k = all k succeed (reliability).

NVIDIA: Production gates care about pass^k (consistency).

Exam: "Works sometimes, not reliably" → gate on pass^k, not pass@k.

🛠️

NAT performance tools

2 terms
NAT ProfilerD3D4D8

Measures tokens, latency, cost at workflow/step/tool granularity to find bottlenecks.

NVIDIA: Pairs with the Sizing Calculator (Profiler finds the hot step; Sizing estimates GPUs).

Exam: "One LLM step is 40% of latency" → Profiler; fixes = smaller model for that step, fewer chunks, caching.

Sizing Calculator (NAT)D3D4

Estimates compute (GPU count, replicas) to hit a throughput/latency target at a given concurrency.

NVIDIA: nat sizing calc; slope-based across concurrency.

Exam: "How many GPUs at 200 concurrent users?" → Sizing Calculator (capacity, not bottleneck-finding).

Section 5 · Fixing what eval found

Tuning & customization

Climb the ladder, never jump it: prompt → RAG → LoRA/QLoRA → SFT → DPO/RLHF, each ~10× the prior. Most "we need fine-tuning" is solved by prompting or RAG. Fine-tuning = skills/behavior; RAG = knowledge/facts.

🪜

The tuning ladder

6 terms
Tuning ladderprompt → RAG → fine-tune → alignD3

Cheapest-fix-first escalation: (1) prompting → (2) RAG → (3) LoRA/QLoRA → (4) full SFT → (5) DPO/RLHF.

NVIDIA: NAT Config Optimizer (nat optimize) = rung 1; NeMo Customizer = rungs 3–5.

Exam: The single most-tested tuning principle. Fine-tuning = skills; RAG = facts.

Prompt engineering / Parameter optimizationD2D3

Shaping behavior via instructions, few-shot, schemas, sampling params — and automating that search vs an eval set.

NVIDIA: NAT Config Optimizer (nat optimize, [config-optimizer] extra) searches prompt wording, temperature, top-p, few-shot, tool descriptions.

Exam: "Tune the agent without training" → parameter optimization (rung 1).

SFT (Supervised Fine-Tuning)D3

Training on labeled input→output pairs, updating weights toward a task/style. Full SFT updates all weights (rung 4).

NVIDIA: A NeMo Customizer job — best quality ceiling, most data/GPU.

Exam: "Skill/format gap at scale with labeled data" — after prompting and RAG plateau.

LoRA / QLoRA (PEFT)D3

Parameter-efficient: train small adapters (~0.1–1% of params). QLoRA quantizes the base to 4-bit to fit larger models on small GPUs.

NVIDIA: A NeMo Customizer technique (rung 3); ~100s–10k examples.

Exam: "Consistent format at scale, labeled data, limited GPU" → LoRA/QLoRA.

DPO (Direct Preference Optimization)D3

Preference alignment trained directly on (chosen, rejected) pairs — no reward model, no RL loop.

NVIDIA: A NeMo Customizer job (NAT harness drives it); the default alignment method.

Exam: "Align tone with preference pairs, minimal infra, no reward-model expertise" → DPO.

RLHF / GRPOD3

RLHF: reward model + RL (PPO), frontier-scale. GRPO: RL comparing a group of rollouts against a reward — no pairs, no reward model.

NVIDIA: NAT's harness supports GRPO via OpenPipe ART.

Exam: "Can auto-score each run (a reward) but no chosen/rejected pairs, want learning from experience" → GRPO.

Section 6 · Safety, guardrails & compliance

Safety & guardrails

Guardrails enforce at runtime; auditing/red-teaming catch issues before ship. The biggest distractors here are the rail categories (it's five, and "topical" isn't one of them) and the exact Safety NIM names.

🛡️

NeMo Guardrails & the rails

4 terms
GuardrailD9

A runtime mechanism that monitors/validates/modifies inputs, outputs, or actions to enforce policy.

NVIDIA: NeMo Guardrails, configured in Colang.

Exam: Runtime control (vs pre-deployment auditing). Know what each rail catches.

NeMo GuardrailsD9

Open-source toolkit for programmable guardrails on LLM apps — five rail categories, Colang config.

NVIDIA: library mode (pip install nemoguardrails, CPU) or microservice mode (container, NGC, GPU for Safety NIMs). Core class LLMRails, configured by RailsConfig.

Exam: Tests rail categories + Colang. Know the library/microservice split.

The five rail categoriesinput · dialog · output · retrieval · executionD9D6

Input (screen/alter user input, mask PII) · Dialog (govern flow / prompting — includes topical control) · Output (fact-check, redact) · Retrieval (screen RAG chunks) · Execution (validate tool/action I/O).

NVIDIA: All implemented as Colang flows.

Exam: "Topical" is NOT a category — it's a dialog-rail behavior. Retrieval rails are the forgotten fifth.

ColangD9

NVIDIA's DSL for guardrail flows, actions, event handlers (Colang 2.0 = Python-like).

NVIDIA: The config language for NeMo Guardrails; all rails are Colang flows.

Exam: "What configures NeMo Guardrails behaviors?" → Colang.

🚨

Safety NIMs, attacks & auditing

5 terms
Safety NIMs (NemoGuard family)D9

NIMs that classify content for safety: Llama 3.1 NemoGuard 8B Content Safety (harm taxonomy), Topic Control (stay on approved topics), NemoGuard Jailbreak Detect.

NVIDIA: Integrated as parallel input/output rails; reasoning-capable variants exist.

Exam: Use exact names; map each NIM to its job (harm / topic / jailbreak).

Jailbreak / Prompt injectionD9D2

Attacks that bypass safety/system-prompt constraints — role-play, prompt extraction, encoding tricks, injected instructions in tool/RAG content.

NVIDIA: Detected by NemoGuard Jailbreak Detect + input rails; tested by garak.

Exam: Jailbreak (bypass safety) vs injection (untrusted content hijacks) — know which rail/NIM defends each.

PII & redactionD9D8

Data identifying an individual; redaction masks it before storage/transmission/display.

NVIDIA: Guardrails PII detection (GLiNER, Presidio, Private AI) on input/output rails; NAT telemetry redaction masks PII in traces.

Exam: Two redaction contexts — guardrail output rail (user-facing) and observability traces.

garakGenerative AI Red-teaming & Assessment KitD9

NVIDIA's open-source LLM vulnerability scanner — probes jailbreaks, injection, leakage, toxicity, hallucination ("nmap for LLMs").

NVIDIA: Apache-2.0; integrates with NeMo Guardrails; NAT also ships red-teaming middleware.

Exam: "NVIDIA tool to automate LLM red-teaming" → garak (garak tests; guardrails enforce).

Safety for Agentic AI (Blueprint)+ pre-deployment auditingD9

NVIDIA Blueprint improving safety, security & privacy at build, deploy & run stages — the home for pre-deployment auditing (bias/safety/policy).

NVIDIA: Supersedes the older standalone "Safety Blueprint"; works with garak + Guardrails + Safety NIMs.

Exam: Pre-deployment auditing (before ship) vs runtime guardrails (live) — both in the safety lifecycle.

Section 7 · Deployment & inference

Deployment, serving & inference

Inference efficiency is a named exam lever. Know Dynamo (disaggregated prefill/decode + KV-cache reuse), NIM serving, and the three structured-output mechanisms.

⚡

Inference framework & decoding

5 terms
NVIDIA DynamoD4

Datacenter-scale, low-latency distributed inference framework for generative/reasoning models.

NVIDIA: Disaggregated serving (separate prefill/decode), KV-cache-aware routing, dynamic GPU scheduling, KV offload. Backends: TensorRT-LLM, vLLM, SGLang.

Exam: "Scale inference, separate prefill/decode, reuse KV cache" → Dynamo.

Disaggregated servingprefill / decodeD4

Split inference into compute-bound prefill (process prompt) and memory-bound decode (generate), on independently scaled workers.

NVIDIA: A core Dynamo capability; KV cache moves prefill→decode over RDMA.

Exam: "Why separate prefill and decode?" → different resource profiles → independent scaling.

KV cache / Prefix cachingD4

Cached attention K/V for processed tokens; prefix caching reuses the cache for a shared prompt prefix to skip recomputation.

NVIDIA: Dynamo KV-aware routing reuses prefixes — big for agents that resend the same system prompt.

Exam: "Many requests share a long system prompt — cut redundant compute" → prefix/KV reuse.

TensorRT-LLMD4D7

NVIDIA's library for optimized LLM inference on GPUs (kernel fusion, quantization, in-flight batching).

NVIDIA: The engine inside many NIMs; a NIM/Dynamo backend (alongside vLLM/SGLang).

Exam: The "why is the NIM fast?" optimization layer.

Structured / guided decodingnvext.guided_jsonD2D4

Constraining the decoder so output must conform to a schema/grammar/regex/choice — guaranteed parse.

NVIDIA: On NIM, the nvext extension: guided_json, guided_regex, guided_choice, guided_grammar (xgrammar default). Pattern: Pydantic → schema → guided_json → validate.

Exam: Three mechanisms — prompt-only (no guarantee), JSON mode (valid JSON, any shape), guided decoding (schema-guaranteed). On NIM it's nvext.guided_json.

Section 8 · Interoperability

Protocols — MCP & A2A

One line settles most protocol questions: MCP connects an agent to tools; A2A connects an agent to other agents. NAT is both client and server for each.

🔌

MCP

1 term
MCP (Model Context Protocol)D2D7

Open standard ("USB-C for AI tools") for dynamic tool discovery + invocation: a server exposes tools/resources/prompts; a client connects at runtime to discover and call them.

NVIDIA: NAT is both MCP client (consume remote tools) and MCP server (publish functions, incl. FastMCP). Transports: stdio (local) / streamable HTTP (remote).

Exam: "Discover tools at runtime across a boundary" → MCP. Three primitives: tools/resources/prompts. Third-party servers = injection surface.

🤝

A2A

1 term
A2A (Agent-to-Agent)D1D7

Protocol for inter-agent messaging — typed payloads, task delegation, status tracking, agent discovery between deployed agent services.

NVIDIA: NAT as A2A client (delegate to remote agents) and server (publish a workflow as a discoverable agent), built on the official A2A Python SDK with auth.

Exam: "Build a team of distributed agents that delegate to each other" → A2A. Don't swap with MCP.

Section 9 · Run & monitor

Observability & operations

Production agents are debugged through traces. Know how NAT exports telemetry and how to localize a multi-step failure to a layer.

📡

Telemetry & monitoring

4 terms
OpenTelemetry (OTel)D8

Open standard for collecting distributed traces, metrics, logs.

NVIDIA: NAT exports via OTel to Phoenix, Weave, Langfuse, with workflow→step→tool granularity.

Exam: "Standard, vendor-neutral way to trace an agent" → OpenTelemetry.

Tracing / spansD8D3

A trace = one full agent run; spans = nested timed segments (workflow → step → tool).

NVIDIA: NAT emits spans at workflow/step/tool level (needs stable IDs).

Exam: Tracing reconstructs the trajectory to debug where a run failed.

Drift detectionD8D3

Monitoring for distribution shift between eval data and live traffic, or degrading output quality over time.

NVIDIA: An online-eval signal; drifted failures feed back into the golden set (Data Flywheel).

Exam: "Quality silently degrades after deploy as inputs change" → drift; fix loop = detect → add to golden set → re-tune.

Failure-mode distributionD3D8

Breakdown of where failures occur — planning, tool call, or synthesis.

NVIDIA: Key diagnostic artifact from trajectory eval + tracing.

Exam: "Most failures at the tool-call step" → harden tool schemas/args, not the model.

Section 10 · Cognition, planning & memory

Cognition, memory & planning

Memory turns a chatbot into something that remembers you across sessions; test-time compute spends inference to think harder. Know NAT's memory backends and the hallucination link to faithfulness.

🧩

Memory & deliberation

5 terms
Memory (agent)D5

Retaining/recalling info across interactions — history, preferences, learned facts, task context.

NVIDIA: NAT memory system with multiple Object Store backends + the Automatic Memory Wrapper.

Exam: Short-term (in-context) vs long-term (persisted, retrieved on demand).

Object Store (NAT)D5

NAT's storage backend abstraction for memory/state.

NVIDIA: Backends: in-memory (dev), MySQL (queryable), Redis (low-latency), S3 (large objects).

Exam: "Low-latency persistent memory" → Redis; "queryable" → MySQL; "dev" → in-memory; "big artifacts" → S3.

Automatic Memory Wrapper (NAT)D5

Transparently adds memory to any agent — intercepts I/O, stores to an Object Store, injects relevant memories later.

NVIDIA: Zero-code memory; less control than manual management.

Exam: "Add memory without rewriting the agent" → Automatic Memory Wrapper (convenience vs control).

Test-time computeD5D3

Spending extra inference compute (multiple calls, deliberation, candidate selection) to raise quality.

NVIDIA: Embodied in NAT's Reasoning Agent (plan / generate / select).

Exam: "Spend more at inference for a better answer" → test-time compute (an inference-time lever, not training).

HallucinationD5D3D9

Generating content not supported by context/training — fabricated facts/citations stated as real.

NVIDIA: Mitigated by RAG (grounding), Guardrails fact-check output rail, faithfulness eval.

Exam: In RAG, hallucination = low faithfulness → a generation fix (assuming retrieval was good).

Section 11 · Where third-party sheets go wrong

Source corrections — verified against current NVIDIA docs

These are the places where popular cram sheets (and the reference glossary we mined) are stale or wrong. Each was checked against current NVIDIA documentation — they make excellent distractors, so learn the correct version.

FIX 1 "NAT has seven agent types"

Current NAT docs list eight: ReAct, Reasoning, ReWOO, Responses API Agent, Router, Parallel Executor, Sequential Executor, Tool Calling. The Parallel Executor is the one most often dropped from "seven types" lists.

FIX 2 "The five rails are input / output / dialog / topical / execution"

The five real categories are input, dialog, output, retrieval, execution. "Topical" is not a top-level rail — topic control is a dialog-rail behavior (or the Topic Control NIM). Retrieval rails are the category most people forget.

FIX 3 Generic "Content Safety NIM / Jailbreak NIM" names

The exact products are Llama 3.1 NemoGuard 8B Content Safety, Llama 3.1 NemoGuard 8B Topic Control, and NemoGuard Jailbreak Detect. NVIDIA's red-team tool is garak, and the current safety Blueprint is Safety for Agentic AI. Use the exact names; generic ones are weaker distractors.

⚠ Version-dependent: Current NeMo Retriever NIMs are llama-nemotron-embed-1b-v2 / llama-nemotron-rerank-1b-v2; they supersede the older llama-3.2-nv-embedqa-1b-v2 / nv-rerankqa-1b-v2 (2026-05-18 deprecation). The roles (embedding NIM, reranking NIM) are stable; verify the specific model name on the current model card before quoting it.
Section 12 · Recall drills

Glossary quiz — can you produce the term?

Definitions are easy to recognize and hard to recall. These drills force production — given a scenario, name the exact term/product.

Q1You need an agent that fans out three independent data lookups at once and merges them. Which NAT agent type — and why do "seven agent types" cheat sheets miss it?Show answer ▾Hide ▴
Parallel Executor.There are eight NAT agent types; the Parallel Executor is the one most often dropped from third-party lists. Sequential runs stages in order; Router dispatches to one specialist; Parallel runs many at once.
Q2A colleague says NeMo Guardrails has five rails: input, output, dialog, topical, execution. What's wrong, and what's the real fifth?Show answer ▾Hide ▴
"Topical" is not a category — it's a dialog-rail behavior. The real five are input, dialog, output, retrieval, execution.Retrieval rails screen/alter retrieved RAG chunks before they prompt the LLM — the forgotten category.
Q3Your RAG answers are fluent but contradict the retrieved docs; context relevance 0.9, faithfulness 0.55. Which metric named this, and which half do you fix?Show answer ▾Hide ▴
Faithfulness (groundedness) caught it; fix the generation side.Context is relevant (0.9), so retrieval is fine — the generator is hallucinating past good context. Tighten grounding/prompt or judge-gate regeneration.
Q4You can auto-score whether each agent run hit its goal (a reward) but have no chosen/rejected pairs, and want the agent to improve with experience. Which finetuning path, and via which NAT integration?Show answer ▾Hide ▴
GRPO via OpenPipe ART (NAT finetuning harness).GRPO compares a group of rollouts against the reward — no preference pairs, no reward model. DPO would apply only if you had (chosen, rejected) pairs.
Q5Two protocols: one lets your agent discover and call a partner team's tools at runtime; the other lets your agent delegate a subtask to a separate deployed agent. Name each.Show answer ▾Hide ▴
Tools → MCP; agent → agent → A2A.MCP = Model Context Protocol (tools/resources/prompts, stdio or HTTP). A2A = Agent-to-Agent (delegation/discovery, official A2A Python SDK). NAT is client and server for both.
Q6A 6-step agent shows 88% per-step accuracy yet ~46% end-to-end success. The team wants a bigger model. Name the phenomenon and the better levers.Show answer ▾Hide ▴
Compound error: 0.88⁶ ≈ 0.46.A bigger model barely moves per-step accuracy. Better: cut the number of steps, harden the weakest step (clearer tool schema, grounding), add a verification step. Use the failure-mode distribution to find the bleeding step.
Q7You must guarantee a NIM returns output that always parses against a Pydantic schema — not "please reply in JSON." What mechanism, and the exact NIM request field?Show answer ▾Hide ▴
Guided (constrained) decoding via extra_body={"nvext": {"guided_json": schema}}.Prompt-only gives no guarantee; JSON mode gives valid-but-any-shape JSON; guided decoding (xgrammar backend) constrains tokens to the schema. Pattern: Pydantic → model_json_schema() → guided_json → model_validate_json().
Q8Leadership wants to cut inference cost on a high-volume tool-calling agent using traffic you already log, without losing accuracy. Which NVIDIA Blueprint, and which microservices does it wire?Show answer ▾Hide ▴
Data Flywheel Blueprint.NAT collects logs (tagged by workload_id), NeMo Evaluator scores candidates, NeMo Curator builds datasets, NeMo Customizer fine-tunes/distills a smaller model that still clears the bar — promoted after human review.
Q9Profiler shows one LLM step is 40% of latency and most tokens. Name two fixes — and which tool answers "how many GPUs at 200 concurrent users?"Show answer ▾Hide ▴
Fixes: route that step to a smaller/faster model, or retrieve fewer chunks (+reranker)/cache. GPU count → NAT Sizing Calculator.Profiler finds the hotspot; Sizing Calculator does capacity planning. Both are system-level.
Q10Pairwise GPT-judge says prompt v2 wins 70%; flipping the answer order makes v1 win 65%. Name the bias and the standard fix.Show answer ▾Hide ▴
Position bias.Fix: judge both orderings (swap A/B) and count only consistent verdicts, or randomize positions. Also watch verbosity and self-preference bias.
