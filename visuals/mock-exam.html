Section 2 · Exam strategy

Timing & tactics — ~2 minutes per question

~65 questions in 120 minutes ≈ ~110 seconds each. You can't derive answers from first principles; you need instant pattern recognition. Hold ~10 minutes back for a review pass.

⏱️Budget the clock

65 × 110s ≈ 119 min. Target ~100s/Q on pass 1, keep ~10 min for pass 2.

🚩Flag & move

Anything you'd spend >2 min on, flag it. Never sink 4 min into a 1.5%-of-score question while easy ones wait.

🔁Pass 2 unlocks pass 1

A later question often jogs the concept an earlier one needed. Return to flags with fresh context.

✅Never blank

No negative marking on MCQ. Eliminate, then guess — a flagged blank is a guaranteed zero.

Section 3 · Wording traps

The three trap words that flip the answer

These are wording traps, not knowledge traps. A single word inverts the correct choice. Most "I knew this and still missed it" errors live here.

TRAP 1 NOT / EXCEPT / LEAST — you're picking the wrong one

Three options are correct facts; the answer is the false / worst / odd-one-out. Fix: underline the negative word before reading the options. Example: "Which is NOT a reason to go multi-agent?" → the right answer is the single-agent scenario.

TRAP 2 BEST / MOST appropriate / FIRST — many work, one is purpose-built

Multiple options technically function; choose the one designed for exactly this scenario. "Could work" loses to "built for it." On FIRST questions (error rate jumped, latency spiked) the answer is almost always observe / diagnose before you act — read traces, then change prompt/model. Changing the model first is the classic wrong answer.

TRAP 3 True-but-irrelevant distractor

A statement that is factually correct but doesn't answer this question. Fix: re-read the stem — correctness ≠ relevance. "MCP makes tools faster" can be true and still be the wrong answer to "what problem does MCP solve?"

Section 4 · The elimination engine

The NVIDIA-first heuristic

For any "which tool / how would you" question with 4 plausible options, eliminate by layer and lifecycle, then prefer the most specific NVIDIA primitive.

①Wrong layer?

NIM = inference. NAT = orchestration. They never swap. "NIM orchestrates the agent" is always wrong; so is "NAT runs the weights."

②Wrong lifecycle?

Eval tools (Eval Framework, Profiler, Sizing Calc) aren't deployment tools; deployment servers aren't safety tools.

③Two left? Be specific

Prefer the named primitive over a generic best practice: execution rail > "add validation"; NeMo Curator > "preprocess"; NIM Operator > "write K8s YAML".

④Tie-break

A documented NVIDIA feature beats an inferred best practice. The exam tests NVIDIA-documented behavior.

D1

Agent Architecture & Design

15% · 8 questions
Q1D1 · BESTA support system handles 5 distinct request categories, each needing different tools and prompts. One ReAct agent with all tools, or a Router dispatching to 5 specialists?Show ▾Hide ▴
Router + 5 specialists. One agent holding all tools bloats the prompt and tool-selection accuracy degrades past ~8–12 tools. The Router classifies intent once (cheap) and dispatches to a focused specialist that scales and is evaluated independently.Distractors: "single ReAct is simpler" is true but loses on accuracy at scale; "Sequential Executor" is wrong because the categories are alternatives, not a fixed pipeline.Concept link: D1 — Router pattern & single-vs-multi-agent.
Q2D1 · selectMinimize total LLM calls on a predictable multi-tool task where mid-run adaptation is rarely needed. Which agent type, and what do you give up?Show ▾Hide ▴
ReWOO. It plans all tool calls up front in one reasoning pass, then executes (independent tools in parallel within a dependency level) without re-reasoning between steps — far fewer LLM calls than ReAct's per-step loop. You give up adaptivity: a fixed plan can't course-correct on surprises.Distractors: ReAct minimizes errors on unpredictable tasks, not LLM calls; the Reasoning agent still centers a single function; Sequential Executor isn't an LLM-reasoning pattern.Concept link: D1/D5 — ReWOO plan-then-execute vs ReAct interleave.
Q3D1 · BESTA research agent queries flaky databases where ~10% of calls fail. ReAct or ReWOO?Show ▾Hide ▴
ReAct. Reasoning after each step lets it retry, reformulate, or switch sources on failure. ReWOO's pre-baked plan breaks on the first unexpected failure with no recovery path.Distractor: "ReWOO is cheaper" is the bait — cheaper-but-brittle loses when 1-in-10 steps fail.Concept link: D1 — adaptivity vs efficiency tradeoff.
Q4D1 · compareThe architectural difference between a Sequential Executor and a Router agent?Show ▾Hide ▴
Sequential Executor = fixed ordered pipeline (output of N → input of N+1, every step always runs, no decision logic). Router = examines input, classifies, dispatches to one of several agents (a routing decision). Sequential for "extract→transform→validate→store"; Router for "this type here, that type there."Distractor: "both make routing decisions" is false — only the Router decides; that's the trap.Concept link: D1 — executor vs router.
Q5D1/D6 · whyAI-Q uses two-stage retrieval (embed → rerank). Why is the rerank stage worth it when you already have a good embedding model?Show ▾Hide ▴
Bi-encoder embedding scores query and document independently — fast over millions of docs but coarse. The reranker is a cross-encoder that processes the query and each candidate together, capturing fine-grained interaction the bi-encoder misses. Two-stage = bi-encoder recall + cross-encoder precision on top-N.Distractor: "rerank replaces the embedder" is wrong — it follows it on a small candidate set.Concept link: D6 — bi-encoder vs cross-encoder, AI-Q architecture.
Q6D1 · NOTWhich is NOT a reason to prefer multi-agent over single-agent?Show ▾Hide ▴
Choices: (a) large, easily-confused tool set · (b) sub-tasks need independent scaling · (c) sub-tasks need different prompts/models · (d) a simple linear pipeline with one tool. → Answer is (d): that's the textbook single-agent (or Sequential Executor) case; multi-agent there is over-engineering.Why the others are valid drivers: (a)(b)(c) all genuinely justify multi-agent — the NOT-trap hides the single-agent case among them.Concept link: D1 — when multi-agent earns its complexity.
Q7D1 · recallName the four phases of the agent loop and where the LLM "thinks."Show ▾Hide ▴
Perceive → Reason → Act → Observe, looping. The LLM does the Reason (decide next action) and interprets the Observe (tool result) to decide loop-or-finish.Distractor: "the LLM executes the tool" is wrong — the runtime executes; the LLM only chooses the tool and reads the result.Concept link: D1 — perceive/reason/act/observe.
Q8D1/D4 · BESTMulti-agent system: a Router that only classifies intent, and a Code-Analysis agent that finds subtle security bugs. Best LLM-size allocation?Show ▾Hide ▴
Small model (~8B) for the Router, large model (~70B) for Code Analysis. Classification is easy and high-traffic — a big model there wastes GPU and adds latency. Nuanced reasoning needs the big model.Distractor: "use 70B everywhere for consistency" is the trap — uniformity isn't the goal; fit-for-purpose is.Concept link: D1/D4 — model-size-per-agent.
D2

Agent Development

15% · 8 questions
Q9D2 · conceptIn NAT, what is a Function Group, and what do Per-User Functions add?Show ▾Hide ▴
A Function Group is a named, reusable bundle of tools you assign to an agent as a unit instead of listing tools individually. Per-User Functions make specific tools available based on the authenticated user (admin gets delete_record, a regular user doesn't) — role-based tool access without forking the config per role.Distractor: "function groups are middleware" conflates two different concepts.Concept link: D2 — function registry, function groups, per-user functions.
Q10D2/D7 · conceptWhat problem does NAT's MCP client solve that a static tool registry does not?Show ▾Hide ▴
Dynamic, runtime tool discovery across organizational boundaries. A static registry binds tools at build time; an MCP client connects to an MCP server at runtime, discovers tools, and invokes them — even tools added after deploy. NAT can also be an MCP/FastMCP server, publishing its workflow as a tool for other agents.Distractor: "MCP makes tools faster" is true-but-irrelevant to the actual problem MCP solves.Concept link: D2/D7 — MCP client vs server.
Q11D2 · recallName the main NAT middleware uses (function pre/post-invoke hooks), one line each.Show ▾Hide ▴
NAT exposes a function-middleware interface (pre/post-invoke hooks) with several documented uses: Caching (reuse responses for identical/similar inputs — FAQ agents), Defense (input validation / attack detection before the LLM — prompt-injection screening), Logging (structured audit of I/O, tool-calls, timing — compliance), and Red-teaming (auto-generate adversarial inputs pre-deploy). Reason about the purpose of each rather than memorizing a fixed count — the exact catalog evolves across releases.Distractor: "rate-limiting" / "load-balancing" are not the documented agent-middleware uses — generic-infra bait.Concept link: D2 — middleware stack.
Q12D2/D7 · BESTA team has a working LangChain agent and wants NVIDIA eval, guardrails, and prod deployment without rewriting. Fastest path?Show ▾Hide ▴
Wrap it with the NAT plugin for LangChain. The plugin adapts the framework interface into a NAT workflow, gaining NAT's deployment servers, Evaluation Framework, Profiler, middleware, and Guardrails — zero logic rewrite. Plugins exist for LlamaIndex, CrewAI, AutoGen, Semantic Kernel, etc.Distractor: "rebuild it as a native NAT ReAct agent" ignores the explicit "without rewriting" constraint.Concept link: D2/D7 — plugin / framework interop.
Q13D2/D3 · conceptWhat does the NAT Agent Hyperparameter (Parameter) Optimizer tune, and how?Show ▾Hide ▴
It searches YAML-configurable parameters — system-prompt wording, temperature, top-p, max tokens, # few-shot examples, tool descriptions — by running the agent against an eval dataset and keeping the config that maximizes a target metric.Distractor: "it fine-tunes model weights" is wrong — it tunes config, not weights.Concept link: D2/D3 — config optimization vs fine-tuning.
Q14D2 · NOTWhich NAT authentication mechanism is NOT standard? (API Key, OAuth2, Bearer, HTTP Basic, MCP accounts, SAML federation)Show ▾Hide ▴
"SAML federation" is not in the documented NAT auth set. The documented mechanisms are API Key, OAuth2, Bearer, HTTP Basic, and MCP accounts.Why a trap: SAML sounds enterprise-correct, but the exam tests the documented list — always confirm against current docs, as auth surfaces evolve.Concept link: D2 — auth mechanisms.
Q15D2/D9 · conceptYou want tool calls to always have well-formed parameters. What's the generation-time lever (vs the runtime guardrail)?Show ▾Hide ▴
NAT structured outputs / function-calling with JSON schemas constrain the LLM to emit schema-valid tool calls at generation time. The complementary runtime lever is a Guardrails execution rail that validates the call before it fires. Defense in depth = both.Distractor: picking only the execution rail misses that the stem asks for the generation-time control.Concept link: D2/D9 — structured output + execution rail.
Q16D2 · recallWhat must a minimal NAT agent YAML declare?Show ▾Hide ▴
Agent type, model/LLM endpoint, tools (functions or a function group), and a system prompt — optionally middleware, parameters, memory, guardrails. The YAML is the declarative spec of what the agent does; nat builds the runnable workflow from it.Distractor: "the YAML contains the model weights" is nonsense bait — weights live in the NIM, not the config.Concept link: D2 — YAML workflow structure.
D3

Evaluation & Tuning

13% · 6 questions
Q17D3 · diagnoseRAG system: faithfulness 0.95, answer-relevance 0.60. Where is the fault?Show ▾Hide ▴
Retrieval. High faithfulness = the answer faithfully reflects the retrieved passages; low relevance = those passages don't address the question. The generator is honestly summarizing the wrong context. Fix retrieval (embeddings, reranking, query reformulation).Distractor: "raise temperature / change the prompt" attacks the generator and makes it worse.Concept link: D3 — faithfulness vs relevance diagnosis.
Q18D3 · FIRSTProfiler: retrieval 0.5s, rerank 0.8s, LLM generation 9.0s, guardrails 1.5s (p95 ≈ 12s). Optimize first?Show ▾Hide ▴
LLM generation (75% of latency). Options: smaller NIM if quality allows, enable Dynamo prefix-caching, trim max output tokens, or add NIM replicas if queuing. Guardrails (1.5s) is the secondary target (parallelize the safety NIMs).Distractor: "optimize retrieval" — it's already fast; touching it is wasted effort. The FIRST-trap rewards attacking the dominant cost.Concept link: D3 — Profiler hotspot → fix.
Q19D3 · conceptWhat does trajectory evaluation catch that output-only evaluation misses?Show ▾Hide ▴
Process errors: wrong tool, wrong order, or wasteful calls — even when the final answer is right ("right answer via the wrong path"), or right but via 10 calls when 3 would do. You define expected trajectories and compare actual ones.Distractor: "trajectory eval measures answer correctness" — that's output eval; trajectory eval scores the path.Concept link: D3 — trajectory vs output eval.
Q20D3 · compareProfiler vs Evaluation Framework — which answers which question?Show ▾Hide ▴
Profiler = "how fast / how expensive?" (tokens + latency at workflow/step/tool level, cost). Evaluation Framework = "how good?" (faithfulness, relevance, accuracy, trajectory correctness). Use both: eval guards quality, profiler guards resources.Distractor: "Profiler measures faithfulness" swaps the two — a classic conflation trap.Concept link: D3 — profiler vs eval split.
Q21D3 · BESTNo labeled test set exists for a RAG agent. Best practical evaluation bootstrap?Show ▾Hide ▴
(1) Generate synthetic Q&A from the corpus (passage → questions it answers). (2) Use the Evaluation Framework with LLM-as-Judge for reference-free metrics — faithfulness and relevance need no labels. (3) Use synthetic pairs as pseudo-ground-truth for recall@k. (4) Later, have experts label a real-query subset.Distractor: "wait until you have human labels" stalls forever — the cold-start trap.Concept link: D3 — LLM-as-judge, synthetic eval.
Q22D3 · toolCut inference cost on a high-volume tool-calling agent using traffic you already log, without losing accuracy. Which NVIDIA pattern?Show ▾Hide ▴
The Data Flywheel Blueprint. It mines tagged production logs, builds eval/finetune datasets, finds a smaller distilled model that still clears the accuracy bar (NeMo Evaluator + Customizer), and promotes it after human review.Distractor: "just switch to a smaller model" skips the measure-it-still-passes step the flywheel exists to provide.Concept link: D3 — Data Flywheel loop.
D4

Deployment & Scaling

13% · 6 questions
Q23D4 · selectNAT offers FastAPI, MCP, FastMCP, and A2A servers. When A2A over FastAPI?Show ▾Hide ▴
When agents must talk to other agents via the Agent-to-Agent protocol — typed inter-agent messages, task delegation/lifecycle, agent discovery, automatic trace propagation. FastAPI is for external clients over REST/WebSocket. Typical shape: FastAPI gateway on the Router (external), A2A between specialists (internal).Distractor: "A2A is just FastAPI with extra steps" ignores the typed-contract / discovery / lifecycle features.Concept link: D4 — server type selection.
Q24D4 · conceptWhat does the NIM Operator automate that raw Kubernetes YAML does not?Show ▾Hide ▴
GPU discovery/allocation, model caching across pod restarts (PVCs), LLM-aware health probes (model-load / inference readiness, not just TCP), and inference-load-based scaling. It encapsulates NVIDIA's operational know-how for serving on K8s.Distractor: "it orchestrates the agent logic" — no, that's NAT; the Operator manages NIM lifecycle.Concept link: D4 — NIM Operator.
Q25D4 · conceptWhich Dynamo optimization is especially valuable for agents?Show ▾Hide ▴
Prefix caching — agents reuse the same long system prompt every request, so caching the KV-cache for that shared prefix avoids recomputation. Dynamo also does dynamic batching, disaggregated prefill/decode serving, and multi-node inference.Distractor: "Dynamo replaces NAT orchestration" is a layer error — Dynamo optimizes inference, not orchestration.Concept link: D4 — Dynamo for agents.
Q26D4 · feasibilityA 70B model on 2×H100 handles ~20–40 concurrent. You have 8×H100, need 500 concurrent on a 70B model. Feasible?Show ▾Hide ▴
No. Do the GPU math: a 70B model in FP16 (~140 GB weights) needs ~2×H100 (80 GB each) per NIM instance, so 8×H100 ÷ 2 = only 4 instances → 4 × (20–40) ≈ 80–160 concurrent, far short of 500. Options: drop to an 8B/13B model (one instance fits a single H100, so 8×H100 could clear 500+), raise GPUs to ~24–32×H100, add aggressive caching, or queue with timeouts. Validate with the NAT Sizing Calculator.Distractor: "yes, just add replicas" ignores that replicas need GPUs you don't have.Concept link: D4 — sizing, GPU planning.
Q27D4 · conceptWhy is request-count autoscaling insufficient for agent endpoints?Show ▾Hide ▴
Agent requests have wildly variable resource cost — one complex multi-step request can consume 100× the tokens/GPU of a simple one — so request count doesn't track load. Scale on token throughput / GPU queue depth / utilization instead.Distractor: "agents don't get requests" / "K8s can't count requests" are nonsense baits.Concept link: D4 — variable-length scaling.
Q28D4/D7 · selectWhich NAT server exposes your agent as a callable MCP tool for other MCP-compatible agents?Show ▾Hide ▴
The MCP server (or FastMCP server runtime). FastAPI serves humans/apps; A2A is agent-to-agent; the MCP server publishes the workflow as an MCP tool.Distractor: "A2A" is the near-miss — A2A is for agent peers, MCP is for tool exposure.Concept link: D4/D7 — MCP/FastMCP publishing.

Cognition, Planning & Memory

10% · 5 questions
Q29D5 · conceptWhat is Test-Time Compute in the NAT Reasoning Agent, and what does it cost?Show ▾Hide ▴
Spending extra inference compute to improve quality: generate multiple candidates, plan/deliberate, select the best — a deliberation step instead of single-pass generation. It trades latency and tokens (often 3–5×) for accuracy on hard reasoning.Distractor: "it fine-tunes at inference" — no weights change; it's extra forward passes.Concept link: D5 — test-time compute.
Q30D5 · matchMatch each NAT Object Store backend to its best fit: in-memory, Redis, MySQL, S3.Show ▾Hide ▴
In-memory: dev/test, ephemeral sessions (fast, volatile). Redis: production session memory needing low-latency reads + survival across restarts (+ TTL). MySQL: long-term, queryable, auditable, backed-up memory (compliance). S3: large artifacts not needing low-latency random access (archival).Distractor: "use S3 for hot session reads" — wrong latency profile; S3 is archival, not hot-path.Concept link: D5 — object store selection.
Q31D5 · NOTWhat is the Automatic Memory Wrapper, and when should you NOT use it?Show ▾Hide ▴
It transparently adds memory to any agent — intercepting I/O, storing relevant info, injecting retrieved memories into context — with no logic change. Don't use it when: memory needs are highly specific/structured; injected memories would blow the context budget and evict important context; you need fine-grained retain/forget control; or the agent is intentionally stateless.Distractor: "never use it in production" overstates — it's fine when its automatic policy matches your needs.Concept link: D5 — automatic memory wrapper.
Q32D5 · recallReWOO's three phases?Show ▾Hide ▴
Planning (create the full plan + all tool calls), Execution (run each step; independent tools can run in parallel within a dependency level), Solution (synthesize gathered evidence into the final answer).Distractor: "Perceive/Reason/Act" is the generic agent loop, not ReWOO's named phases.Concept link: D5 — ReWOO internals.
Q33D5 · BESTA conversational agent must remember preferences across sessions with sub-millisecond reads. Which memory config?Show ▾Hide ▴
NAT memory on the Redis object store + the Automatic Memory Wrapper. Redis persists across restarts, reads in <1ms, supports TTL for aging out stale memories; the wrapper handles store-on-turn and retrieve-into-context. Add MySQL as a secondary store only if you need a queryable audit trail.Distractor: "in-memory store" loses everything on restart — the cross-session requirement fails.Concept link: D5 — cross-session memory.

Q34D6 · recallName the AI-Q / NVIDIA RAG Blueprint retrieval stack (embed, rerank, vector store) and the default generation model class.Show ▾Hide ▴
Embed: llama-nemotron-embed-1b-v2. Rerank: llama-nemotron-rerank-1b-v2. Vector store: Milvus (GPU-accelerated via cuVS). Default generation in the current AI-Q Blueprint is a Nemotron Super–class model (e.g. Llama-3.3-Nemotron-Super-49B-v1.5), swappable to any NIM-compatible model.Currency: NeMo Retriever NIM 1.13.0 renamed these to the Nemotron brand — older strings call them llama-3.2-nv-embedqa-1b-v2 / llama-3.2-nv-rerankqa-1b-v2; either may appear, but the current names are the llama-nemotron-* ones (embed keeps 8192-token, multilingual, Matryoshka dims). Distractor: "FAISS / OpenAI embeddings" — not the NVIDIA stack.Concept link: D6 — AI-Q components.
Q35D6 · whenWhen does sparse retrieval (BM25) beat dense (embeddings)?Show ▾Hide ▴
On rare exact tokens — error codes, identifiers, SKUs ("error NV-4231") the embedder never learned vectors for; very short 1–2 word queries with little semantic signal; specialized vocab outside the embedder's training. Production answer for technical corpora: hybrid (dense + sparse, fused) usually beats either alone.Distractor: "dense always wins" is the over-generalization trap.Concept link: D6 — dense/sparse/hybrid.
Q36D6 · conceptWhat does cuVS do for Milvus, and when does the GPU pay off?Show ▾Hide ▴
cuVS is NVIDIA's CUDA library for GPU-accelerated approximate-nearest-neighbor search; in Milvus it replaces CPU ANN (CPU-HNSW/IVF), parallelizing distance computation across many concurrent queries. It pays off at high concurrency (e.g. >50 QPS with tight SLAs) or very large indexes; for single-digit-QPS dev, CPU-HNSW is fine.Distractor: "cuVS makes single queries asymptotically faster" — the win is throughput under concurrency, not single-query big-O.Concept link: D6 — Milvus + cuVS.
Q37D6 · tool5M documents; CPU preprocessing takes 72 hours. Which NVIDIA tool, and what does it accelerate?Show ▾Hide ▴
NeMo Curator (GPU-accelerated): fuzzy dedup (MinHash LSH), language ID/filtering, quality scoring, extraction/cleaning — typically 10–100× over CPU. Pair with NIM embedding endpoints (batch) and Milvus+cuVS indexing for an end-to-end GPU ingestion pipeline.Distractor: "NeMo Customizer" is fine-tuning, not data prep — the Curator/Customizer swap is the deliberate trap.Concept link: D6 — NeMo Curator.
Q38D6 · MOST-LIKELYUsers report outdated answers from a RAG policy agent. Most likely cause?Show ▾Hide ▴
The index wasn't refreshed with recent policy updates — stale index is the most common cause of stale answers. Fix: re-run ingestion; long-term, automate freshness.Distractors: "LLM is outdated," "embedding model too small," "wrong chunking" are all possible but far less likely than an unrefreshed index — the MOST-LIKELY trap rewards the operational cause.Concept link: D6 — index freshness / operations.

Run, Monitor & Maintain

5% · 2 questions
Q43D8 · recallName NAT's three tracing levels and the redaction processor's job.Show ▾Hide ▴
Workflow level (whole request: total latency, total tokens, status), Step level (each reasoning step / ReAct iteration), Tool level (each tool call: name, params, response, time) — drill from "request took 8s" to "the DB query tool took 5s." A redaction processor masks/removes sensitive data (API keys, PII, proprietary content) from traces before export.Distractor: "two levels (request + tool)" omits the step level — a count trap.Concept link: D8 — trace granularity + redaction.
Q44D8 · diagnoseAn agent occasionally loops, calling the same tool repeatedly. Which metric detects it, and which backends does NAT export to?Show ▾Hide ▴
A sudden spike in steps-per-execution flags looping (token cost rises too, but as a secondary signal). NAT exports OpenTelemetry traces to Phoenix, Weave, Langfuse (and raw OTel).Distractor: "CPU utilization" doesn't isolate a reasoning loop — it's a true-but-irrelevant signal.Concept link: D8 — loop detection, exporters.
D9

Safety, Ethics & Compliance

5% · 3 questions
Q45D9 · recall ⚠Name the five NeMo Guardrails rail types and place "topic control" correctly.Show ▾Hide ▴
The five are: input, dialog, retrieval, output, execution. Topic control is a use of dialog rails (or the Topic Control NIM) — there is no separate "topical rail" type. Input rails screen/alter user input; retrieval rails filter RAG chunks; output rails screen/alter the response; execution rails validate tool/action calls.Distractor: any option listing "topical" as one of the five canonical types is wrong by current docs — the single most common exam-trap in this domain.Concept link: D9 — five rail types.
Q46D9 · conceptHow do execution rails provide agentic security? Give a concrete blocked action.Show ▾Hide ▴
They intercept a tool call before it executes and validate name, parameters, and context against policy — so even a prompt-injected or buggy LLM can't fire an unauthorized call. Concrete: an injection makes the LLM emit delete_record("*"); the execution rail validates the id is a real UUID (not a wildcard), checks the user's permission, and/or requires human approval — blocking the mass delete.Distractor: "an output rail would catch it" — output rails act on text, not the action; the action layer is the execution rail.Concept link: D9 — execution rails.
Q47D9 · BESTPublic-facing agent facing adversarial users: Colang-only rules, or Colang plus Safety NIMs?Show ▾Hide ▴
Colang rules + Safety NIMs (Content Safety, Jailbreak Detection, Topic Control). Colang patterns handle topic/flow/simple filtering; the ML-based Safety NIMs catch novel phrasings, encoding tricks, and multi-turn manipulation that pattern-matching misses. Colang-only is acceptable only for internal, trusted, structured-input agents.Distractor: "Safety NIMs alone, drop Colang" loses dialog/flow control — defense-in-depth wants both.Concept link: D9 — safety NIMs + rails.
D10

Human-AI Interaction & Oversight

5% · 3 questions
Q48D10 · primitiveAn execution rail decides a tool call needs human approval. How does NAT deliver the request to the human in real time?Show ▾Hide ▴
A NAT WebSocket interactive workflow sends the pending action to the connected client and waits for approve/reject (bidirectional, real-time). On approve the rail lets the call proceed; on reject it blocks and returns feedback.Distractor: "email notification" / "write to a DB and poll" aren't the NAT real-time primitive — the specific-primitive trap.Concept link: D10 — WebSocket approval gate.
Q49D10 · BESTAn agent does both low-risk password resets and high-risk Active Directory edits. Best oversight architecture?Show ▾Hide ▴
Risk-proportional: auto-execute the low-risk resets; gate the AD edits behind a human approval via a Guardrails execution rail (+ WebSocket). Log everything via logging middleware for the audit trail.Distractors: "approve everything" kills throughput on safe actions; "approve nothing" is reckless on AD; "log but never gate" provides no real control — all three are BEST-trap foils.Concept link: D10 — risk-proportional oversight.
Q50D10 · NOTWhich is NOT a NVIDIA-documented HITL primitive (vs an inferred design pattern)?Show ▾Hide ▴
Choices: WebSocket interactive workflows · execution-rail approval gates · per-user functions/auth · a built-in escalation-routing dashboard. → Answer: the escalation-routing dashboard. NVIDIA provides the primitives; escalation paths, approval queues, and feedback loops are design patterns you build on those primitives, not shipped components.Why it matters: this is the exam's weakest-coverage domain — prefer the answer anchored to a documented primitive over an inferred pattern.Concept link: D10 — primitives vs patterns.
Section 6 · Cross-domain

Integrative challenges — the two capstones

The exam's hardest questions cross domain boundaries. These two scenarios are the shape of those questions. The drill: walk every domain and name the NVIDIA primitive that implements your choice. Do both end-to-end and you can answer any cross-domain item.

⚠ Why these matter: a guardrails question may need deployment knowledge; a memory question may need evaluation knowledge. Studying modules in isolation answers isolated questions; studying them as a system answers the harder cross-boundary questions that separate passing from failing.
