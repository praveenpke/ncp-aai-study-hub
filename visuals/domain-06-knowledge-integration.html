Knowledge Integration & Data Handling

The knowledge plumbing behind every grounded agent: build a RAG pipeline end-to-end — ingest → chunk → embed → index → retrieve → rerank → generate — pick the right index and search mode for the workload, handle messy real-world data (PDFs, tables, PII, staleness), and know when plain vector RAG isn't enough and you reach for GraphRAG.

Section 1 · Why & how to think about it

Mental model — a research library

Agents are only as good as the knowledge they can reach. This domain is the layer where most RAG quality problems actually live — and where the exam asks you to pick the right component/strategy for a scenario: which chunking strategy, which ANN index, dense vs hybrid, when to add a reranker, which NVIDIA NIM does X. The library analogy maps every stage to a role.

📚Acquisitions = Ingestion

Books arrive in every format; a cataloger (NV-Ingest) extracts the usable content — including the tables and figures a lazy cataloger would skip.

🗃️Catalog cards = Chunking

One card per book is too coarse; one per sentence is noise. Cards that follow chapter boundaries work best.

🔖Card catalog = Embed + Index

Organized so the librarian jumps to the right shelf without scanning every card — that's ANN.

🧑‍🏫Two librarians = Dense + Sparse

Dense understands what you mean; sparse/BM25 matches the exact words (codes, IDs, names). Hybrid asks both and merges (RRF).

🔬Subject expert = Reranker

Takes 40 candidate books, actually skims each against your question, hands you the best 5. Too slow for the whole library — perfect for a shortlist.

🕸️Reference librarian = GraphRAG

Knows how books relate — "the author of X was funded by Y, acquired by Z" — answering questions no single shelf contains.

Collection maintenance = data handling: redact the donor records (PII) before shelving, and re-catalog when new editions arrive (freshness) — a catalog of last year's library only answers last year's questions.

The whole pipeline at a glance

The stages are independently tunable and independently measurable — evaluate retrieval (recall@k, NDCG) separately from generation (faithfulness), per Domain 3.

The two-stage retrieval mantra

① Retrieve wide & cheap

Bi-encoder ANN over the whole corpus. Encode query and passages separately, compare vectors — scalable, precomputable.top-40…100 candidates

② Rank narrow & expensive

Cross-encoder scores each query+passage pair jointly — accurate but per-pair, so only over the shortlist.→ keep top 3–5

Almost every "improve precision" question resolves to adding the second stage; almost every "the answer is missing entirely" question resolves to fixing the first.
Section 2 · Core concepts

The RAG pipeline, stage by stage

Each stage has a job and a signature failure mode if done badly. Mapping a symptom to the responsible stage is the single most-tested skill in this domain.

Chunking strategies

Chunk size is a retrieval parameter, not a context-window maximization game. No universal winner exists — the meta-lesson the NVIDIA study and 2025–26 benchmarks all agree on is: sweep chunking on your own corpus with retrieval metrics.

StrategyHow it splitsProsConsWhen
Fixed-sizeEvery N tokens/chars (e.g. 512 tokens, 10–15% overlap)Trivial, predictable, fastCuts mid-thought; ignores structureBaseline; homogeneous prose
RecursiveTry separators in order (\n\n → \n → . → ) until pieces fitRespects boundaries while honoring a size budget — the production defaultStill structure-blind for tables/headersDefault starting point (~512 tokens)
SemanticEmbed sentences; split where adjacent-sentence similarity drops below a thresholdTopically coherent chunksExpensive (embeds at ingest); tiny fragments; inconsistent gainsTopic-drifting docs; only if measured to help
Structure-awareSplit on real structure: markdown headers, sections, pagesPreserves semantic units; NVIDIA found page-level best on average (0.648, lowest variance)Needs structured input; unit sizes varyManuals, filings, wikis
Overlap (10–20%) prevents answers from being severed at chunk boundaries — at the cost of index size and duplicate-ish results. It does not fix bad chunk sizing. Hierarchical / parent-child chunking (retrieve small, hand the LLM the bigger parent) is the common production resolution of the precision-vs-context tension.

Embeddings, vector DBs & ANN indexes

A bi-encoder encodes query and passage independently; cosine/dot similarity ≈ relevance. Key properties: dimension (Matryoshka lets you truncate for cheaper search), max input tokens, domain coverage, and asymmetry — retrieval embedders use distinct input_type="query" vs "passage" modes. Using the same mode for both silently degrades retrieval.

Distance metric — a property of the model, not a free knob. Cosine (magnitude-invariant) is the safe text default; dot/inner-product equals cosine when vectors are L2-normalized (normalizing forces every vector to length 1, so the only thing left to compare is direction — exactly what cosine measures) and IP is what many embedders optimize; L2/Euclidean is rarely right for normalized text embeddings. Mismatching it quietly tanks recall: e.g. a model trained on cosine but indexed with raw L2 will rank a long, generic chunk (big magnitude) above a short, on-topic one, because Euclidean distance is fooled by vector length. Milvus metric_type="COSINE"/"IP"/"L2"; pgvector vector_cosine_ops / vector_ip_ops / vector_l2_ops.

GPU-accelerated search: cuVS & the Milvus GPU index family NVIDIA-specific

cuVS (CUDA Vector Search), part of RAPIDS, is the library Milvus calls for GPU indexes (it also accelerates FAISS's GPU path). The GPU index types: GPU_CAGRA, GPU_IVF_FLAT, GPU_IVF_PQ, GPU_BRUTE_FORCE.

⚡CAGRA = GPU-native HNSW

CUDA-Accelerated Graph index — the flagship GPU graph index the AI-Q/RAG Blueprints use. ~12× faster build, ~4.7× faster search than CPU HNSW; up to ~100× throughput under heavy batched load.

📊Throughput, not latency

GPU indexes win on high QPS / large query batches and fast builds — for a single low-traffic query a tuned CPU HNSW can match them. The trigger is QPS, not just corpus size.

🧠GPU-memory bound

The index must (largely) fit in GPU memory, with weak OOM protection — oversubscribing can crash the query node. Hybrid pattern: build with CAGRA on GPU, serve as HNSW on CPU (CAGRA graphs beat native HNSW latency at >512D).

Hybrid search: dense + sparse with RRF

Dense vectors capture meaning but fumble exact tokens (part numbers, error codes, names, acronyms). Sparse/lexical scoring (BM25: term-frequency × inverse-document-frequency, length-normalized) nails exact matches but has zero notion of synonyms. Hybrid runs both and fuses the ranked lists.

➕Reciprocal Rank Fusion (RRF)

RRF score(d) = Σ 1/(k + rank_i(d)), smoothing constant k = 60 by default.

RRF only uses ranks, never raw scores — so it needs no normalization across incomparable scales (cosine vs BM25). That's exactly why it's the default fuser (Milvus RRFRanker).

⚖️Weighted scoring

The alternative — WeightedRanker — normalizes and linearly combines scores when you want to bias toward one channel (e.g. 0.7 dense / 0.3 sparse).

Metadata filtering — the third leg. Scalar predicates (source == "2025_10K" and year >= 2024) applied to the vector search. Modern engines pre-filter (constrain the ANN search to matching vectors) rather than post-filter (which can return fewer than k). Filters are also your tenancy/ACL enforcement point — security by filter, not by hoping the LLM ignores other tenants' chunks.

Reranking: cross-encoders for precision

A cross-encoder feeds query and candidate passage through the model together, letting attention compare them token-by-token — far more accurate than bi-encoder cosine, but it must run per query-passage pair at query time, so it can never scan the whole corpus. Hence the two-stage pattern.

🎯What it buys you

  • Higher answer accuracy and lower generation cost (fewer, better chunks → fewer prompt tokens)
  • The natural merge point when retrieving from multiple sources
  • Pattern: retrieve wide (top 40–100) → rerank → keep top 3–5

🧩NVIDIA reranking NIM

llama-nemotron-rerank-1b-v2 — multilingual companion to the embedding NIM (renamed in NeMo Retriever NIM 1.13.0 from llama-3.2-nv-rerankqa-1b-v2; VL variant llama-nemotron-rerank-vl-1b-v2). Predecessor: nv-rerankqa-mistral-4b-v3 (a truncated, LoRA-tuned Mistral with a binary relevance head).

Latency is the price — budget reranking only over the shortlist.

Data handling: parsing, PII, freshness

The unglamorous but highest-leverage stage. You cannot retrieve what was destroyed at parse time, you cannot un-leak PII once it's embedded, and a vector index is just a snapshot that answers last year's questions unless you update it.

📄Multimodal extraction named NIMs

NV-Ingest (NeMo Retriever extraction): splits docs into pages, classifies elements, uses specialized NIMs to extract text, tables (as markdown), charts, infographics into a uniform JSON/DataFrame schema — then optionally chunks, embeds, uploads to Milvus in one fluent pipeline.

  • nemoretriever-page-elements-v2 — detects text/table/chart boxes (layout)
  • nemoretriever-table-structure-v1 — recovers rows/cols → markdown
  • nemoretriever-graphic-elements-v1 — chart/figure elements
  • PaddleOCR — the OCR engine reading characters from scans/images (Apache-2.0; the Blueprint's named document-OCR component)

Tables as structured markdown retrieve dramatically better than whitespace soup. Detectors find layout; PaddleOCR reads characters — a scanned table needs both.

🔒PII redaction

Redact at ingestion time, before indexing — once PII is embedded and stored, every retrieval can leak it, and embeddings themselves can be inverted.

Tooling: NeMo Curator's PiiModifier (built on Microsoft Presidio) for pattern/NER detection; LLMPiiModifier / AsyncLLMPiiModifier for LLM-based detection via a NIM. Runtime NeMo Guardrails checks are defense in depth, not the load-bearing control.

🔄Freshness & re-indexing

RAG's whole advantage over fine-tuning is cheap knowledge updates — but only if you actually update.

  • Incremental upsert: content hash + last_modified per doc; on change, delete its chunks by stable ID (doc_id#chunk_n) and re-embed only it
  • Scheduled sync: cron diff, or event-driven webhook on doc save
  • Full re-embed only for: changing embedding model, chunking strategy, or dimension — version the model in index metadata
Stale-data symptom: confidently outdated answers citing superseded docs — filterable by effective_date metadata until re-ingestion lands. The most common production RAG complaint isn't hallucination — it's confidently citing the superseded policy.

NeMo Curator — the GPU data-prep stage more than PII

PII redaction is just one modifier. NeMo Curator is a GPU-accelerated curation toolkit (RAPIDS cuDF + Dask, ~10–100× CPU throughput) that cleans the corpus between raw extraction and chunking — distinct from NeMo Retriever (which serves embed/rerank/extract) and Milvus (storage/search).

🧬Deduplication

Exact (MD5), fuzzy (MinHash + LSH, near-dups), semantic (embedding clusters). Stops the same content being embedded 5× and biasing retrieval.

🌐Language ID

Detect & filter/route documents by language before embedding.

🧹Quality filtering

Heuristic + classifier-based removal of boilerplate, garbled OCR, spam.

🔒PII redaction

PiiModifier (Presidio) / LLMPiiModifier (NIM) — one stage among many. Also: domain classification & transforms.

Exam separation: "Which NVIDIA tool deduplicates / quality-filters / language-filters the corpus at scale?" → NeMo Curator. "Which serves the embedding & reranking models?" → NeMo Retriever. Don't conflate the prep stage with the serving stage.

GraphRAG & structured knowledge

Vector RAG retrieves isolated chunks by similarity — it fails on multi-hop questions ("Which suppliers of our acquired subsidiaries are affected by the new regulation?") where the connecting facts live in relationships across documents, and on global/aggregate questions ("What are the main themes across this corpus?") where the answer is a synthesis, not a passage.

🕸️GraphRAG (Microsoft)

At indexing time, an LLM extracts entities and relationships from chunks into a knowledge graph; community detection (Leiden) clusters it hierarchically; an LLM writes community summaries per level.

  • Global search maps over community summaries (corpus-wide themes)
  • Local search expands from matched entities to neighbors + source chunks (entity-centric multi-hop)

Knowledge graphs also serve agents as structured tools (Cypher/SPARQL over Neo4j) — text2query alongside similarity search.

💸The cost catch & LazyGraphRAG

LLM-extracting a graph over a large corpus is expensive (oft-cited early-2024 figure: ~$33k to index one dataset), and the graph must be maintained as docs change.

LazyGraphRAG (Microsoft, late 2024) defers the expensive work to query time — indexing cost comparable to plain vector RAG (~0.1% of full GraphRAG's) with comparable global-search quality — the pragmatic default when graph queries are occasional.

Decision: vector RAG for "find the passage"; GraphRAG/KG for "connect the facts"; hybrid (graph + vector) in production systems that need both.

AI-Q Blueprint RAG architecture two tiers

The exam expects you to recognize NVIDIA's reference RAG stack as a layered Blueprint and tell the two tiers apart: the foundational RAG Blueprint answers a single grounded question; the AI-Q Research Assistant wraps it in an agentic deep-research loop.

🏗️RAG Blueprint (foundation)

The production retrieve→rerank→generate pipeline, by stage:

  • NV-Ingest (page-element / table / graphic detectors + PaddleOCR)
  • optional NeMo Curator prep (dedup / quality / PII)
  • Embedding NIM llama-nemotron-embed-1b-v2
  • Vector DB — Milvus (+ cuVS) or Elasticsearch
  • hybrid retrieval → Reranking NIM llama-nemotron-rerank-1b-v2
  • NIM LLM generation (grounded + cited)

The embed+rerank pair is tuned together for high BEIR/TechQA accuracy, multilingual + cross-lingual.

🔬AI-Q Research Assistant (agentic layer)

Extends the RAG Blueprint into a deep-research agent, orchestrated by NeMo Agent Toolkit (NAT):

  • ① draft a report plan
  • ② search internal RAG corpus + web (Tavily)
  • ③ write a draft
  • ④ reflect on gaps → issue follow-up queries
  • ⑤ finalize with a source list

The canonical "retrieval-in-the-reasoning-loop" reference (ties to §8 agentic-RAG).

Version-dependent — don't memorize SKUs. The Blueprints' generation LLMs and embedding model versions refresh often (course-era: Llama 3.3 Nemotron Super 49B / Llama 3.3 70B generation, Llama 3.2 NV EmbedQA/RerankQA 1B retrieval, ~2×H100 80GB self-hosted; current builds moved to Nemotron-3 / Nemotron embed-VL and default to API-Catalog hosting with no local GPU requirement). Section 5 · Gotchas

Exam traps & gotchas

The eighteen traps this domain plants. Most are symptom→stage mapping or "X is obsolete / X everywhere" overreach.

01"Bigger chunks = better"

Oversized chunks dilute the embedding (many topics → mushy vector) and stuff noise into the prompt. Chunk size is a retrieval parameter tuned by recall/precision.

02Overlap fixes quality

Overlap prevents an answer span from being split across chunks; it does not fix bad sizing and inflates index size/cost. Different tool for a different job.

03Reranker can fix recall

A cross-encoder only reorders what retrieval returned. Missing from top-50 → fix embeddings/chunking/hybrid/k. Present but buried → reranker.

04Bi- vs cross-encoder inversion

Bi-encoder: encode query & doc separately, compare vectors, scalable (first stage). Cross-encoder: encode the pair jointly, accurate but per-pair (rerank). Can't cross-encode the whole corpus — it runs per pair at query time.

05input_type asymmetry

NeMo embedders are asymmetric: passages input_type="passage", queries "query". Same mode for both silently hurts retrieval. LangChain's embed_documents/embed_query handles it.

06RRF uses scores

RRF uses ranks, not scores — BM25 and cosine live on incomparable scales. "Fusion without normalization" → RRF (k=60); "weighted bias toward one channel" → weighted scoring.

07BM25/sparse is obsolete

Dense embeddings whiff on exact identifiers, acronyms, rare terms. "Users search by error code/SKU and dense misses" → add hybrid, not a bigger embedder.

08HNSW vs IVF mix-up

HNSW: graph, best recall/latency, memory-hungry, incremental-friendly. IVF: clustered lists, cheaper build/memory, needs training data. "Streaming inserts + high recall" → HNSW; "huge static corpus, tight memory" → IVF(+PQ).

09FAISS is a database

FAISS is a library — no server, replication, or DB-grade CRUD/metadata filtering. "Production multi-tenant vector store" → Milvus/pgvector-class, not raw FAISS.

10Swap embedder, embed only new docs

Vectors from different models (or dims) aren't comparable; a mixed index returns garbage quietly. Model swap = full corpus re-embed + re-index, versioned.

11Fine-tune for fresh facts

Knowledge updates are a re-indexing operation in RAG — that's the whole economic argument for RAG over fine-tuning. (Domain 3 crossover, still tested here.)

12PII redaction at generation time

Too late: the PII is already in the index and every retrieved context. Redact during ingestion (Curator/Presidio), enforce access via metadata filters, runtime rails as defense in depth.

13Vector RAG for multi-hop

Similarity retrieves chunks that look like the question; multi-hop answers live in chains of facts that individually don't. "Joining facts across documents" → GraphRAG/KG (local search for entity-hops, global for themes).

14GraphRAG everywhere

Also a trap: full GraphRAG indexing is LLM-expensive and high-maintenance. Single-hop factual QA → plain vector RAG; occasional graph queries → LazyGraphRAG-style deferred approaches.

15Pre- vs post-filtering

Post-filtering (search, then discard) can leave fewer than k results and wastes compute; production engines pre-filter. Metadata filters are your ACL boundary, not the LLM's discretion.

16Curator vs Retriever confusion

NeMo Curator = GPU data prep (dedup, language ID, quality filter, PII) before chunking; NeMo Retriever = embedding + reranking + extraction NIMs that serve retrieval. "Which dedups / quality-filters the corpus?" → Curator. "Which serves embed/rerank?" → Retriever. And Curator ≠ just PII.

17GPU search = "only for huge data"

The trigger is QPS/throughput, not raw corpus size; cuVS/CAGRA help at 1M+ vectors under high query pressure, and the index must fit in GPU memory. "Sub-10ms at very high QPS" → GPU (cuVS/CAGRA); "occasional latency-sensitive queries, tight GPU budget" → CPU HNSW (optionally CAGRA-built).

18PaddleOCR is the whole extraction

Extraction is a team: detector NIMs (page-elements, table-structure, graphic-elements) find where & recover structure; PaddleOCR reads the characters. A scanned table needs both — layout detection to find it, OCR to read its cells.

Section 6 · Practice

Scenario drills & quick quiz

Read each scenario, decide the fix before you open the answer. These are the exact shape of Domain 6 questions: a symptom, and you map it to the responsible stage.

D1Support-bot users query by product SKU and error code; dense retrieval returns related-but-wrong articles. Recall on keyword-heavy queries is poor; recall on natural-language queries is fine. Fix?show ▾hide ▴
Hybrid search: add a sparse/BM25 channel and fuse with RRF.Exact identifiers are lexical signals dense embeddings can't represent; a bigger embedder won't fix it.
D2Recall@50 is 0.92 but answers are mediocre; the right chunk is usually around rank 20–40, and the LLM receives 50 chunks. Cheapest fix?show ▾hide ▴
Add the reranking NIM (cross-encoder, e.g. llama-nemotron-rerank-1b-v2, formerly llama-3.2-nv-rerankqa-1b-v2) over the 50 candidates, keep top-5.The precision-stage textbook case — improves accuracy and cuts prompt tokens.
D3A financial-analyst agent must answer "How did supplier risks discussed in the 2023 10-K evolve through the 2025 filings?" spanning entities and relationships across many documents. Vector RAG keeps returning isolated, similar-sounding paragraphs. Architecture change?show ▾hide ▴
GraphRAG / knowledge-graph retrieval: extract entities/relations at indexing, use local (entity-expansion) search for the multi-hop chain.LazyGraphRAG-style deferred indexing if cost is a concern.
D4Your corpus is scanned PDFs with dense tables and charts. Text extraction "works" but retrieval over table content is near-zero. First fix?show ▾hide ▴
Fix extraction, not chunking: run ingestion through NV-Ingest / NeMo Retriever extraction so tables become structured markdown and charts become described elements before chunking/embedding.You cannot retrieve what was destroyed at parse time.
D5Compliance flags that customer SSNs appear in retrieved contexts shown to agents. The team proposes a prompt instructing the LLM to omit PII. Correct response?show ▾hide ▴
Reject prompt-level mitigation as primary control; redact PII at ingestion (NeMo Curator PiiModifier/Presidio) and re-index the affected collection, enforce scope via metadata filters, keep Guardrails-style output checks as a secondary layer.Once PII is embedded and stored, every retrieval can leak it.
D6You upgrade from nv-embedqa-e5-v5 to llama-nemotron-embed-1b-v2 and only embed new documents with the new model into the same collection. Retrieval quality craters. Why?show ▾hide ▴
Mixed embedding spaces: vectors from different models aren't comparable, so nearest-neighbor results are meaningless across the boundary.Re-embed the entire corpus with the new model into a fresh (versioned) index, then cut over.
D7Docs change daily; nightly full re-embedding of 10M chunks is blowing the GPU budget. Better pattern?show ▾hide ▴
Incremental upsert: hash each source doc, on change delete its chunks by stable ID prefix and re-embed only that doc; schedule a periodic diff sync.Reserve full re-embeds for model/chunking changes.
D8A 5M-doc Common-Crawl-style dump: heavy near-duplicates, mixed languages, lots of low-quality pages. Retrieval surfaces the same content five times and confidence is low. Which NVIDIA tool, and what operations?show ▾hide ▴
NeMo Curator (the GPU data-prep stage, before chunking): fuzzy dedup (MinHash + LSH) to collapse near-duplicates, language ID to keep target languages, quality filtering to drop boilerplate — all GPU-accelerated via RAPIDS cuDF/Dask.A data-preparation fix, not a retrieval-architecture one — reranking/hybrid won't undo a 5×-duplicated corpus. Don't answer "NeMo Retriever" (that serves embed/rerank).
D9Serve vector search at very high QPS over 8M vectors with strict throughput SLOs, and you have spare H100 capacity. Which index / library?show ▾hide ▴
Milvus GPU index via cuVS — GPU_CAGRA (GPU-native graph index): built for batched high throughput (~12× build, ~4.7× search vs CPU HNSW, up to ~100× throughput). Watch GPU memory (index must fit).If queries were occasional & latency-sensitive instead, a tuned CPU HNSW (optionally CAGRA-built, HNSW-served) is more economical — GPU's edge is throughput, not single-query latency.
D10You need an agent that produces a sourced research report: plan it, pull from the internal RAG corpus and the live web, then revise based on gaps. Which NVIDIA blueprint, and what orchestrates it?show ▾hide ▴
AI-Q Research Assistant Blueprint, extending the foundational RAG Blueprint with a deep-research loop — plan → search (RAG corpus + Tavily web) → write → reflect-on-gaps → re-query → finalize with citations — orchestrated by NeMo Agent Toolkit (NAT).Plain RAG Blueprint answers one grounded question; AI-Q adds the agentic multi-step report workflow on top.

Quick quiz one-liners

Q1Which fuser needs no score normalization across cosine and BM25, and why?show ▾hide ▴
RRF (Reciprocal Rank Fusion, k=60).It combines ranks (1/(k+rank)), never raw scores — sidestepping the incomparable cosine vs BM25 scales entirely.
Q2Streaming inserts plus a high-recall requirement — which ANN index?show ▾hide ▴
HNSW.Best recall/latency and handles incremental inserts well; IVF needs representative training data and degrades on distribution shift.
Q3Which chunking strategy did NVIDIA's study find best on average, with the lowest variance?show ▾hide ▴
Page-level (document-structure-aware) chunking — 0.648 accuracy, lowest variance.But benchmarks disagree; the exam-safe answer is "sweep on your own corpus with retrieval metrics."
Q4Production multi-tenant vector store with metadata filtering and replication — FAISS or Milvus?show ▾hide ▴
Milvus (or pgvector-class).FAISS is a library, not a database — no server, no replication, no DB-grade CRUD/metadata filtering.

🔗1 · Minimal end-to-end RAG embedding NIM + FAISS + reranking NIM + generation

from langchain_nvidia_ai_endpoints import ChatNVIDIA, NVIDIAEmbeddings, NVIDIARerank  # reads NVIDIA_API_KEY
from langchain_community.vectorstores import FAISS
from langchain.retrievers import ContextualCompressionRetriever
from langchain_text_splitters import RecursiveCharacterTextSplitter

chunks = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=64).split_documents(docs)
emb = NVIDIAEmbeddings(model="nvidia/llama-nemotron-embed-1b-v2", truncate="END")  # was llama-3.2-nv-embedqa-1b-v2
vstore = FAISS.from_documents(chunks, emb)                      # passages embedded as input_type=passage

retriever = ContextualCompressionRetriever(                     # stage 2: cross-encoder precision
    base_retriever=vstore.as_retriever(search_kwargs={"k": 40}),  # stage 1: wide & cheap
    base_compressor=NVIDIARerank(model="nvidia/llama-nemotron-rerank-1b-v2", top_n=5),  # was ...nv-rerankqa-1b-v2
)
llm = ChatNVIDIA(model="meta/llama-3.3-70b-instruct")
ctx = retriever.invoke("What is the warranty period for model X?")  # query embedded as input_type=query
answer = llm.invoke(f"Answer ONLY from context:\n{ctx}\n\nQ: What is the warranty period for model X?")

What to notice: the whole §2 pipeline in ~12 lines — retrieve wide (k=40) then rerank down to 5 is the two-stage mantra, and NVIDIAEmbeddings/NVIDIARerank are just OpenAI-compatible NIM endpoints behind LangChain classes. The embed/rerank model pair (llama-nemotron-embed-1b-v2 + llama-nemotron-rerank-1b-v2, formerly the llama-3.2-nv-embedqa/rerankqa-1b-v2 names) is NVIDIA's matched multilingual duo.

✂️2 · Chunking comparison recursive vs semantic in a few lines

from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_experimental.text_splitter import SemanticChunker
from langchain_nvidia_ai_endpoints import NVIDIAEmbeddings

recursive = RecursiveCharacterTextSplitter(            # tries \n\n, \n, ". ", " " in order
    chunk_size=512, chunk_overlap=64).split_text(long_text)
semantic = SemanticChunker(                            # splits where adjacent-sentence similarity drops
    NVIDIAEmbeddings(model="nvidia/llama-nemotron-embed-1b-v2"),
    breakpoint_threshold_type="percentile").split_text(long_text)
print(len(recursive), len(semantic))                   # semantic: variable sizes, topical coherence — and embedding cost at ingest

What to notice: recursive enforces a size budget while respecting separator hierarchy (the production default); semantic spends embedding calls during ingestion to find topic boundaries and emits variable-size chunks. Benchmarks disagree on which wins — the exam-safe answer is "measure on your corpus" (§3.2).

🔀3 · Milvus hybrid search dense + BM25 sparse fused with RRF

from pymilvus import MilvusClient, AnnSearchRequest, RRFRanker

client = MilvusClient(uri="http://localhost:19530")
q_text = "RTX 5090 driver error 0x887A0006"
dense_req = AnnSearchRequest(data=[embed(q_text)], anns_field="dense_vector",
                             param={"metric_type": "COSINE"}, limit=20)
sparse_req = AnnSearchRequest(data=[q_text], anns_field="sparse_vector",   # raw text: Milvus 2.5+ computes BM25 internally
                              param={"metric_type": "BM25"}, limit=20)
hits = client.hybrid_search(collection_name="docs",
                            reqs=[dense_req, sparse_req],
                            ranker=RRFRanker(k=60),                        # rank-based fusion, k=60 default
                            limit=5, output_fields=["text", "source"])

What to notice: the error code 0x887A0006 is exactly what dense-only retrieval misses and BM25 nails. RRFRanker fuses by rank (1/(k+rank)), so the incomparable COSINE and BM25 score scales never need normalizing; swap in WeightedRanker(0.7, 0.3) to bias channels instead.

🔄6 · Freshness: incremental upsert content hash + stable chunk IDs

import hashlib

def sync_doc(doc_id: str, text: str, client, emb):
    h = hashlib.sha256(text.encode()).hexdigest()
    prev = client.query("docs", filter=f'doc_id == "{doc_id}"', output_fields=["content_hash"], limit=1)
    if prev and prev[0]["content_hash"] == h:
        return "unchanged"                                        # skip: no re-embedding cost
    client.delete("docs", filter=f'doc_id == "{doc_id}"')          # targeted delete of stale chunks
    rows = [{"id": f"{doc_id}#{i}", "doc_id": doc_id, "content_hash": h,
             "text": c, "dense_vector": v, "embedding_model": "llama-nemotron-embed-1b-v2"}
            for i, (c, v) in enumerate(zip(chunk(text), emb.embed_documents(chunk(text))))]
    client.insert("docs", rows)
    return "reindexed"
# run from a nightly scheduler / on-save webhook; alert if "reindexed" rate spikes

What to notice: the hash check makes re-indexing incremental — only changed docs pay the embedding cost — and stable IDs (doc_id#n) make deletes surgical. Storing embedding_model per row is what makes a future model migration detectable instead of silently mixing vector spaces (trap #10).

🕸️7 · GraphRAG-style multi-hop sketch extract triples, hop the graph, then answer

import networkx as nx

G = nx.DiGraph()
for chunk in chunks:                                      # indexing time: LLM extracts (head, relation, tail)
    for h, r, t in llm_extract_triples(chunk.text):       # e.g. ("AcmeSub", "supplied_by", "VoltCorp")
        G.add_edge(h, t, relation=r, source=chunk.id)

def multi_hop(question: str, hops: int = 2):
    seeds = llm_extract_entities(question)                # query time: link question → graph entities
    sub = nx.ego_graph(G.to_undirected(), seeds[0], radius=hops)   # expand neighborhood (local search)
    facts, chunk_ids = [], set()
    for u, v, d in sub.edges(data=True):                           # only the retrieved subgraph's edges
        facts.append(f"{u} -{d.get('relation','?')}- {v}")         # the relation, not the whole attr dict
        if d.get("source"): chunk_ids.add(d["source"])             # ground back to local source chunks
    return llm.invoke(f"Facts:\n{facts}\n\nSources:\n{fetch(chunk_ids)}\n\nQ: {question}")

What to notice: the answer is assembled from a path of relationships (ego-graph expansion = GraphRAG "local search"), not from chunks that merely resemble the question — that's why this works for multi-hop where vector similarity fails (§3.7). The expensive part is the indexing-time LLM triple extraction, which is exactly what LazyGraphRAG defers to query time.

🐘8 · pgvector: HNSW index + filtered search plain SQL

CREATE EXTENSION IF NOT EXISTS vector;
CREATE TABLE chunks (id text PRIMARY KEY, doc_id text, tenant text,
                     body text, embedding vector(2048));
CREATE INDEX ON chunks USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 200);            -- HNSW defaults; IVFFlat alt: WITH (lists = 1000)

SET hnsw.ef_search = 100;                          -- query-time recall/speed knob
SELECT id, body, 1 - (embedding <=> $1) AS score   -- <=> is cosine distance
FROM chunks WHERE tenant = 'acme'
ORDER BY embedding <=> $1 LIMIT 5;

What to notice: vectors as a column type means filtering, joins, and transactions are just SQL — the pgvector pitch. The §3.3 knobs appear literally: m/ef_construction at build, ef_search at query; switching to IVFFlat is one index swap (lists ≈ √rows, bulk-load before building).

🧹9 · NeMo Curator data-prep pipeline dedup → lang ID → quality → PII, before chunking

from nemo_curator import Sequential
from nemo_curator.modules import FuzzyDuplicatesRemoval     # MinHash + LSH, GPU
from nemo_curator.filters import FastTextLangId, QualityFilter
from nemo_curator.modifiers import PiiModifier

# Runs on RAPIDS cuDF/Dask — composes as ordered stages
curate = Sequential([
    FuzzyDuplicatesRemoval(threshold=0.8),                  # collapse near-duplicates
    FastTextLangId(keep={"en"}),                            # language identification
    QualityFilter(),                                        # drop boilerplate / garbled OCR
    PiiModifier(entities=["PERSON", "EMAIL", "US_SSN"]),    # Presidio-backed PII redaction, pre-index
])
clean_docs = curate(raw_docs)                               # deduped, filtered, redacted
# only NOW: chunk -> embed (llama-nemotron-embed-1b-v2) -> upsert to Milvus

What to notice: this is the stage between raw extraction and chunking — and it's NeMo Curator, not Retriever (trap #16). Fuzzy dedup is MinHash + LSH; all GPU-accelerated for ~10–100× CPU throughput. PII redaction here is the load-bearing control (trap #12); skipping dedup is what returns the same passage 5× (drill D8). Class names illustrative — verify against the current Curator API.

Section 8 · Field notes

What top engineers are saying (2025–26)

The discourse that reframed RAG for the agent era: retrieval wasn't killed by long context, it was promoted into a decision agents make — and the chunking/GraphRAG cost debates settled into "measure your own corpus."

🗣️LightOn RAG is Dead, Long Live RAG

The sharpest rebuttal to the long-context-killed-RAG narrative: retrieval hasn't been replaced, it's been promoted into a conditional decision agents make (whether/what/where/how to retrieve), and the economics are decisive — "RAG is 8–82× cheaper than long context for typical workloads, with better latency." The meeting analogy (dumping a corpus into context = scheduling a ten-person meeting instead of consulting one expert) is the cleanest framing of why context-stuffing doesn't scale.

lighton.ai/…/rag-is-dead-long-live-rag

🗣️Chroma Research "Context Rot" (Jul 2025)

The empirical backbone of the RAG side: across 18 models (GPT-4.1, Claude 4, Gemini 2.5), performance degrades as input context grows even on simple tasks — million-token windows don't deliver million-token comprehension. Why "just put everything in context" fails, and why curated retrieval into a small context remains the move — it also strengthens the case for reranking (fewer, better tokens).

research.trychroma.com/context-rot

🗣️RAGFlow From RAG to Context: 2025 Year-End Review

The synthesis position: RAG is metamorphosing from a fixed retrieve-then-generate pattern into a "context engine" whose core capability is intelligent retrieval feeding context engineering. Naive RAG (embed → top-k → pray) is what actually died in 2025; the retrieval discipline itself got bigger, not smaller — useful framing for why hybrid, reranking, and agentic retrieval loops are the current default rather than exotic add-ons.

ragflow.io/blog/rag-review-2025-from-rag-to-context

🗣️VentureBeat Six data shifts for enterprise AI in 2026

The pro-disruption side, steelmanned: RAG won't disappear in 2026, but for agentic workloads it's increasingly surpassed by contextual/agentic memory — agents that accumulate task state and consult tools (grep-style, SQL-style search) rather than running a vector pipeline per question. The takeaway echoed across the discourse: vector search becomes one tool among several an agent chooses, not the mandatory front door.

venturebeat.com/…/six-data-shifts-2026

🗣️Microsoft Research LazyGraphRAG

The course correction on GraphRAG's cost problem from GraphRAG's own authors: by deferring LLM work to query time, LazyGraphRAG matches full GraphRAG's global-search quality at ~0.1% of indexing cost (and >700× lower query cost than some global-search configs), while beating vector RAG on local and global queries. Knowledge-graph-quality answers no longer require paying the $33k-style upfront build — removing the main practical objection to graph-augmented retrieval.

microsoft.com/…/lazygraphrag

🗣️Agentic-RAG survey + Red Hat "Tool RAG" arXiv 2501.09136 · Nov 2025

The survey formalizes what practitioners converged on: embed retrieval decisions into the reasoning loop — the agent decides when to retrieve, grades what came back, rewrites the query, and re-retrieves on failure (self-correcting retrieval). Red Hat extends the same pattern to tool selection: RAG over the tool registry itself, with experiments where retrieval-based tool selection jumped accuracy from 13% to 43% on large toolsets.
