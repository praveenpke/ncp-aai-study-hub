Hands-On Labs & Platform Setup

The minimum NVIDIA path from zero to a guarded, deployable agent — a numbered, runnable track: account & key → first NIM call → first NAT workflow → first Guardrails rail → [opt] self-hosted NIM → Blueprint. Every mandatory step runs on a CPU laptop with one API key. Each step ships a verified command and an expected result.

  • Cyan packets = your request flowing through — the same OpenAI-shaped call at every hop

Section 1 · Why & how to think about it

Mental model — get badged in, then commission the line

The other domain pages teach the concepts; this one is the runnable on-ramp. NVIDIA's exam is practitioner-shaped — it assumes you've actually touched build.nvidia.com, nat run, and nemoguardrails chat. Do steps 1–4 once and every other page's Code Companion becomes copy-paste runnable.

🪪1 · Get your badge

Account + nvapi- key. Without it nothing swipes. One env var unlocks the whole hosted platform.

🚪2 · Swipe the simplest door

One chat completion proves the key works and teaches the address: integrate.api.nvidia.com/v1.

🛠️3–4 · Wire a machine + its guard

A NAT agent (model + tool + YAML), then a Guardrails rail that inspects what goes in and comes out.

🏭5–6 · In-house & reference

Optionally bring the machine in-house (self-hosted NIM), then study NVIDIA's published assembly line (a Blueprint).

The insight that ties it together (and a favorite exam point) — API stability. The OpenAI-compatible request you send in step 2 to the hosted endpoint is byte-for-byte the same request you'd send to a self-hosted NIM in step 5. You change exactly one thing: the base_url. Write once, deploy anywhere.

Hosted first, local second, self-hosted last

Every mandatory step is CPU-only. GPUs, Docker, NGC, and Kubernetes appear only in the clearly-marked optional step 5 and the optional full-deployment path of step 6.

Section 2 · The numbered track

Six steps, each with a verified command and an expected result

Run them in order. Steps 1–4 are mandatory and CPU-only; step 5 is optional (GPU); step 6 studies a reference architecture and reproduces a tiny version on CPU.

1

Make a build.nvidia.com account + API key

Goal: a working NVIDIA_API_KEY exported in your shell.
CPU onlyMilestone 0→1

Sign in at build.nvidia.com (the NVIDIA API Catalog), open a model card such as meta/llama-3.3-70b-instruct, click Get API Key, and copy the nvapi-… key (shown once). Then:

export NVIDIA_API_KEY="nvapi-your-key-here"   # add to ~/.bashrc or ~/.zshrc, then source it
echo "$NVIDIA_API_KEY"                         # must print nvapi-…  (not blank, no trailing space)
✓
Expected: echo $NVIDIA_API_KEY prints your key starting nvapi-. Artifact: the exported key.
2

First NIM API call

Goal: prove the key works; learn that NIM is OpenAI-API compatible.
CPU onlyMilestone 1

Base URL is https://integrate.api.nvidia.com/v1; chat path /chat/completions.

curl -s https://integrate.api.nvidia.com/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $NVIDIA_API_KEY" \
  -d '{"model":"meta/llama-3.3-70b-instruct",
       "messages":[{"role":"user","content":"What is NVIDIA NIM in one sentence?"}],
       "max_tokens":80,"temperature":0.2}'

Same call in Python — the one NVIDIA-specific line is base_url:

from openai import OpenAI; import os
client = OpenAI(base_url="https://integrate.api.nvidia.com/v1",
                api_key=os.environ["NVIDIA_API_KEY"])
r = client.chat.completions.create(model="meta/llama-3.3-70b-instruct",
        messages=[{"role":"user","content":"What is NVIDIA NIM in one sentence?"}],
        max_tokens=80, temperature=0.2)
print(r.choices[0].message.content, "|", r.choices[0].finish_reason, r.usage.total_tokens)
✓
Expected: a chat completion with message.content, finish_reason:"stop", and a usage.total_tokens count. finish_reason:"length" means you hit max_tokens (truncated). Artifact: saved response + token count.
Which model? (June 2026.) The labs default to meta/llama-3.3-70b-instruct — stable, tool-calling-capable, least likely to break a copy-paste command. Know that the current NVIDIA headline family is Nemotron 3 (Dec 2025 → Computex June 2026): Nano (~30B/3B active, nvidia/nemotron-3-nano-30b-a3b), Super (~100B/10B), Ultra (~500–550B/50–55B — NVIDIA's largest open model, for reasoning/planning/agentic), plus multimodal Nano Omni. Any is a drop-in swap — change only model_name/model. Pick by capability lane, don't hard-code "best model." Catalog id vs HF repo: the hosted model field wants the lowercase catalog id (nvidia/nemotron-3-nano-30b-a3b); Hugging Face weight repos use a longer name with a precision suffix (nvidia/NVIDIA-Nemotron-3-Nano-30B-A3B-BF16/-FP8). Pasting an HF repo name into a hosted call → 404.
3

First NAT workflow (YAML + nat run)

Goal: a NeMo Agent Toolkit agent that calls a tool, defined entirely in YAML.
CPU onlyMilestone 2–3

Package nvidia-nat · import name nat · CLI nat · NAT v1.7 needs Python 3.11–3.13.

python -m venv nat-env && source nat-env/bin/activate   # Windows: nat-env\Scripts\activate
pip install nvidia-nat
nat --version        # logs help banner + version (e.g. 1.7.x) → verifies the install

Create workflow.yml — note the exact current schema (LLM _type: nim + model_name; agent under workflow: with _type: tool_calling_agent, tool_names, llm_name):

functions:
  current_datetime:            # NAT ships built-in example tools
    _type: current_datetime
llms:
  nim_llm:
    _type: nim                 # NVIDIA NIM provider (reads NVIDIA_API_KEY from env)
    model_name: meta/llama-3.3-70b-instruct
    temperature: 0.0
workflow:
  _type: tool_calling_agent    # native function-calling agent
  tool_names: [current_datetime]
  llm_name: nim_llm
  verbose: true
  handle_tool_errors: true     # default: tool exception → ToolMessage, agent self-corrects
  max_iterations: 15           # default cap on tool-call rounds
nat run --config_file workflow.yml --input "What time is it right now?"
✓
Expected: the agent calls current_datetime and answers in natural language; verbose:true shows the tool call → observation → final answer trace. Swap _type: react_agent to see explicit Thought→Action→Observation (Milestone 3). Artifact: workflow.yml + trace.
4

First NeMo Guardrails run (a Colang rail)

Goal: block off-topic input before the LLM is called; demonstrate a refusal.
CPU onlyMilestone 5

Library mode (this step) is pip + key, CPU-only. The second package is what lets Guardrails reach NVIDIA-hosted models:

pip install nemoguardrails langchain-nvidia-ai-endpoints
nemoguardrails --version

config/config.yml:

colang_version: "2.x"          # use Colang 2 flow syntax
models:
  - type: main
    engine: nim                 # NVIDIA-hosted NIM backend
    model: meta/llama-3.3-70b-instruct
rails:
  input:
    flows:
      - check topic             # runs on every user msg, BEFORE the LLM

config/topics.co (Colang 2, example-based intent matching):

define user ask off topic
  "What's a good chocolate cake recipe?"
  "Who won the game last night?"

define flow check topic
  user ask off topic
  bot refuse off topic

define bot refuse off topic
  "I can only help with technology questions — programming, AI, and software engineering."
nemoguardrails chat --config config/
# then type:  What's a good chocolate cake recipe?
✓
Expected: on-topic questions reach the LLM and get answered; off-topic ones are intercepted by the input rail — the LLM is never called — and the bot returns the refusal. Artifact: config/ + guarded-vs-unguarded transcript.
5

[Optional] Self-hosted NIM

Goal: run the same API on your own GPU. No lab needs this.
GPU + DockerOptional

Requires Linux, driver 535+, Docker 24+, NVIDIA Container Toolkit, and an NGC API key (separate from NVIDIA_API_KEY).

# Username is the LITERAL string $oauthtoken
echo "$NGC_API_KEY" | docker login nvcr.io --username '$oauthtoken' --password-stdin

docker run -d --name nim-llm --gpus all -p 8000:8000 \
  -v nim-cache:/opt/nim/.cache -e NGC_API_KEY="$NGC_API_KEY" \
  nvcr.io/nim/meta/llama-3.1-8b-instruct:latest

curl -s http://localhost:8000/v1/health/ready      # → {"status":"ready"} when up

Then call it with exactly the step-2 code, one line changed:

client = OpenAI(base_url="http://localhost:8000/v1", api_key="not-needed-locally")
✓
Expected: /v1/health/ready returns ready, and /v1/chat/completions on localhost:8000 returns the same OpenAI-shaped JSON as the hosted endpoint — proving API parity. Cleanup: docker stop nim-llm && docker rm nim-llm (the nim-cache volume keeps weights).
6

A Blueprint walkthrough — AI-Q Research Assistant

Goal: read a reference architecture; reproduce a minimal RAG pipeline on CPU.
CPU (study)Milestone 6

Blueprints are production-grade reference architectures (not tutorials) at build.nvidia.com/blueprints. The AI-Q Research Assistant is built on the NeMo Agent Toolkit (+ LangChain Deep Agents): an orchestrator classifies intent, a shallow agent gives quick cited answers, a deep agent writes long-form cited reports. Repo: github.com/NVIDIA-AI-Blueprints/aiq. As of June 2026 it's on the v2.x line and its default models moved to the Nemotron 3 family (nvidia/nemotron-3-nano-30b-a3b) with the renamed retriever embedder nvidia/llama-nemotron-embed-vl-1b-v2. The CPU reproduction below keeps the still-live llama-3.3-70b-instruct + nv-embedqa-e5-v5 so it runs anywhere — the architecture lesson is identical.

import os, numpy as np, faiss
from openai import OpenAI
c = OpenAI(base_url="https://integrate.api.nvidia.com/v1", api_key=os.environ["NVIDIA_API_KEY"])
docs = ["NVIDIA NIM serves optimized inference microservices.",
        "RAG grounds LLM answers in retrieved documents to reduce hallucination.",
        "Milvus is an open-source vector DB; FAISS is a local CPU alternative."]
emb = lambda t: np.array([d.embedding for d in
        c.embeddings.create(model="nvidia/nv-embedqa-e5-v5", input=t).data], dtype="float32")
X = emb(docs); faiss.normalize_L2(X)
idx = faiss.IndexFlatIP(X.shape[1]); idx.add(X)            # cosine via inner product
q = emb(["How does RAG reduce hallucination?"]); faiss.normalize_L2(q)
top = [docs[i] for i in idx.search(q, 2)[1][0]]
ans = c.chat.completions.create(model="meta/llama-3.3-70b-instruct",
        messages=[{"role":"system","content":"Answer ONLY from the context; cite it."},
                  {"role":"user","content":f"Context:\n{chr(10).join(top)}\n\nQ: How does RAG reduce hallucination?"}])
print(ans.choices[0].message.content)
✓
Expected: the embedding call reports a 1024-dim vector per doc, FAISS retrieves the RAG/grounding docs as top hits, and the LLM returns a grounded, citation-style answer. Artifact: the reproduction script + architecture/gap-analysis writeup.

Section 4 · Don't fall for these

Exam traps & gotchas

#1 Web UI host vs API host

build.nvidia.com is the web UI; integrate.api.nvidia.com/v1 is the API host. Pointing curl at build.nvidia.com → 404 / HTML. The API base is always https://integrate.api.nvidia.com/v1, then /chat/completions, /embeddings, …

#2 Two keys: NVIDIA_API_KEY ≠ NGC API key

The build.nvidia.com key authenticates hosted API calls. The NGC key (username literally $oauthtoken) authenticates pulling containers from nvcr.io. Mixing them up is the classic step-5 failure.

#3 NAT YAML field names are exact (and have changed)

Current schema: LLM _type: nim with model_name (not type:/model:); the agent lives under workflow: with _type: tool_calling_agent, tool_names: (a list), llm_name:. There is no agents:/entry_agent: block. Validate with python -c "import yaml;yaml.safe_load(open('workflow.yml'))".

#4 YAML is indentation-sensitive — spaces, never tabs

A file that parses but mis-nests (e.g. tool_names not a list) gives an agent that "has no tools" at runtime. Dump the parsed structure to confirm the shape.

#5 Model names are exact and namespaced

meta/llama-3.3-70b-instruct — the meta/ prefix and -instruct suffix are mandatory; dropping either → 404 "model not found". Copy the exact id from the model card.

#6 OpenAI-compatibility is the whole point

A NIM endpoint speaks the OpenAI Chat Completions API — the openai client works by changing only base_url + api_key. The same request hits hosted (integrate.api.nvidia.com/v1) or self-hosted (localhost:8000/v1) NIM unchanged.

#7 Guardrails engine: nim (current) vs nvidia_ai_endpoints (older)

Same backend; pick by version. Both need langchain-nvidia-ai-endpoints. And set colang_version: "2.x" or your Colang 2 .co flows silently parse as Colang 1.

#8 Input rails run before the LLM, output rails after

"Block this off-topic question without paying for an LLM call" = input rail. "Redact PII from the model's answer" = output rail. Knowing which side a rail sits on is a recurring distinction.

#9 nemoguardrails server needs the [server] extra

The base package has no FastAPI/uvicorn; server mode fails with ModuleNotFoundError: fastapi. pip install nemoguardrails[server].

#10 HTTP codes: 401 vs 404 vs 429

401 = bad/missing key. 404 = wrong model id or wrong host. 429 = free-tier rate limit (not a broken key). For 429, back off and retry — don't regenerate the key.

Section 7 · Test yourself

Quiz — quick recall on the platform path

Q1A teammate's curl to https://build.nvidia.com/v1/chat/completions returns HTML and a 404. What's wrong, and what's the correct host?Show answer ▾Hide ▴
They're hitting the web UI, not the API.build.nvidia.com is the browser catalog. The API base is https://integrate.api.nvidia.com/v1, then /chat/completions. Same key, different host.
Q2You can call the hosted endpoint fine, but docker pull nvcr.io/nim/... fails with "unauthorized" — even though NVIDIA_API_KEY is set. Why?Show answer ▾Hide ▴
Wrong key. Pulling containers needs an NGC API key, not the build.nvidia.com hosted-API key.echo "$NGC_API_KEY" | docker login nvcr.io --username '$oauthtoken' --password-stdin — the username is the literal string $oauthtoken.
Q3Your NAT workflow.yml from an old tutorial uses type: nim, model:, and an agents: + entry_agent: block. nat run errors on unknown fields. Fix the schema.Show answer ▾Hide ▴
Current NAT: LLM is _type: nim with model_name; the agent lives under workflow: with _type: tool_calling_agent, tool_names:, llm_name:.There is no agents:/entry_agent: in current NAT. The _type key (not type) names every entity's kind.
Q4You need to block off-topic questions without paying for an LLM call, and separately redact PII from the model's answers. Which rail type for each?Show answer ▾Hide ▴
Off-topic block → input rail (runs before the LLM, so the LLM is never called). PII redaction → output rail (runs after, on the model's response).Input rails: rails.input.flows; output rails: rails.output.flows. Knowing the side a rail sits on is the recurring distinction.
Q5nemoguardrails chat loads your config but dies the moment it tries to answer, mentioning a missing provider. Both the model id and key are correct. Most likely cause?Show answer ▾Hide ▴
Missing langchain-nvidia-ai-endpoints.engine: nim (and the older engine: nvidia_ai_endpoints) need that package to reach NVIDIA-hosted models. The config loads without it, then fails at model call time. Also check colang_version: "2.x" is set.
Q6You have a GPU and want to self-host. After deploying, how much of your hosted-API client code must change to talk to the local NIM, and what's the gotcha?Show answer ▾Hide ▴
One line — the base_url from integrate.api.nvidia.com/v1 to localhost:8000/v1.The NIM API is identical hosted vs self-hosted (OpenAI-compatible). Gotcha: wait for /v1/health/ready to return ready first — the container loads weights into GPU memory on first start (minutes).
Q7Your RAG reproduction throws a vector "dimension mismatch" when adding to the FAISS index. What two things should you check?Show answer ▾Hide ▴
(1) Index dim must equal the embed model's output dim — nv-embedqa-e5-v5 is 1024. (2) Use the same embed model at index time and query time.Indexing with one model and querying with another (or hardcoding the wrong dim) is the classic cause.
Q8"I can't do the labs, I don't have an H100." How much of the 8-milestone path is actually blocked by that?Show answer ▾Hide ▴
Essentially none of the mandatory path. Steps 1–4 and the step-6 study/partial reproduction are CPU-only; days 1–10 of the 14-day path need no GPU.A GPU is only for the optional self-hosted NIM (step 5) and the optional full Blueprint deployment. Hosted-first means one API key reaches ~90% of the platform.
