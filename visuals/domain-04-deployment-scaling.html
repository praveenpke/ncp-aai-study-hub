Section 1 · Why + Mental Model

The Restaurant Chain Analogy

An agent that works in a notebook dies in production: agents fire many LLM calls per task (planning, tool selection, reflection), so inference cost and latency multiply. The exam tests serving layer, parallelism, scaling signal, and rollout strategy — as best-choice-under-constraint scenarios.

🔪TensorRT-LLM

Industrial kitchen equipment. Quantization = pre-prepped ingredients · in-flight batching = cooks start a new dish the instant a burner frees · paged KV cache = labeled storage bins, no wasted shelf space.

👨‍🍳Triton

The kitchen manager. Runs many menus (frameworks) at once, groups similar orders onto one pan (dynamic batching), chains stations into a production line (ensembles).

📦NIM

Franchise-in-a-box. Sealed container with equipment + manager + menu + a standard order window (OpenAI-compatible API). Plug in a GPU and open.

🏢K8s + GPU Operator + HPA

Corporate operations. Opens more branches when queues grow (horizontal scaling); the load balancer is the host directing customers to the shortest line.

🚚Jetson

The food truck. Small, local, offline-capable — physical-world agents.

🏨DGX Cloud

Renting fully staffed restaurants by the hour. Blue-green/canary = trialing a new menu without poisoning every customer at once.

Section 1.5 · The agent-serving tier

NAT Deployment Servers — Orchestration vs Inference

NIM/Triton/TRT-LLM serve the model. NAT (NeMo Agent Toolkit) serves the agent — the orchestration tier holding the agent graph, tool routing, memory and guardrails, which calls the model tier over HTTP. The single most-tested idea: NAT orchestrates (CPU) · NIM infers (GPU) · NAT calls NIM. Different containers, different scaling signals, different cold-start behavior.

Two tiers that scale independently

many LLM calls (fan-out) autoscaling NAT on CPU won't help if NIM is saturated OpenAI HTTP NIM tier · GPU pod · scale vertical→horizontal NIM container optimized serving · batching · quantization HPA on gpu_cache_usage_perc / token throughput / queue depth the real bottleneck — never CPU GPU cluster H100 / A100 Multi-container by nature: NAT + NIM + vector DB + Redis + observability. One agent ≠ one container — collapsing tiers kills independent scaling.

⏳Async job management (Dask)

Agent runs take seconds→minutes. Holding an HTTP connection open invites timeouts, LB disconnects, retry storms. NAT pattern:

  • Client POSTs → server returns a job ID immediately (background-queued)
  • Dask schedules + executes the workflow
  • Client polls status/result (or subscribes via WebSocket)
  • /evaluate is the canonical Dask-backed async job · knobs: scheduler type, job TTL, history retention

🧪Worked example — front end + sync vs async

"A ~90 s research task; other internal agents must also call this researcher as a tool." → Consumed by other agents that are independent services → A2A (or MCP if just called as a tool in-process). 90 s exceeds safe HTTP sync limits → async job: submit → job ID → poll (Dask-backed), not a synchronous REST call with a long timeout (that's the trap answer).

Section 4 · One GPU, maximum speed

TensorRT-LLM

NVIDIA's open-source library for optimized LLM inference: custom attention kernels, kernel fusion, quantization, optimized runtime. Historically: convert checkpoint → trtllm-build a compiled engine per GPU type / precision / TP config (engines are not portable across GPU architectures). Newer releases are PyTorch-architecture-based with a high-level LLM API; integrates with Triton (TRT-LLM backend), NIM, and NVIDIA Dynamo.

Quantization — shrink weights (and sometimes activations)

Less VRAM → more KV-cache room → higher throughput, lower latency.

Paged KV Cache & In-Flight Batching

🗂️Paged KV cache (PagedAttention idea)

Autoregressive decoding reuses K/V of all previous tokens; caching avoids quadratic recompute but VRAM grows with batch × sequence length. Naive allocators reserve max-seq-length contiguous buffers → fragmentation.

Enables larger effective batches + KV block reuse / prefix caching — agents resending long system prompts reuse cached blocks. Related: chunked prefill splits long prompts into chunks so prefill doesn't stall ongoing decodes.

🔄In-flight (continuous) batching

Static/dynamic batching waits for the whole batch to finish — variable LLM output lengths leave GPU slots idle. In-flight batching works at iteration granularity: a finished request is evicted and a new one injected immediately; prefill and generation requests interleave in the same batch.

Result: higher GPU utilization, lower queue wait. The LLM-specific evolution of dynamic batching — Triton dynamic batching = request-level (stateless models); in-flight = token/iteration-level inside TRT-LLM/vLLM.

Speculative decoding

Generation is memory-bandwidth-bound (one token per heavy forward pass). A cheap draft mechanism proposes several tokens; the target model verifies them in one parallel pass, accepting the longest correct prefix — same output distribution, fewer target passes.

  • Techniques: draft-target (separate small model), EAGLE / EAGLE-3, Medusa (extra decoding heads), MTP (multi-token prediction), NGram lookup
  • Measured: up to ~3.55× throughput on Llama 3.3 70B with draft models on HGX H200
  • Best when batches are small / latency-bound; gains shrink at very high batch occupancy
Section 5 · From node to fleet

Scaling: HPA, Parallelism, Load Balancing, Batch Tuning

Horizontal pod autoscaling on GPU metrics — never CPU

CPU/memory utilization is meaningless for GPU inference pods — they show low CPU while fully saturated.

📊Good scaling signals

  • gpu_cache_usage_perc — KV-cache utilization; the canonical NIM HPA metric (e.g., target AverageValue 100m = 10%)
  • num_requests_running / queue depth (concurrency)
  • time_to_first_token_p90 — latency-SLO-driven scaling

⏱️Caveats

  • Downscale stabilization (default 5 min) to avoid flapping
  • GPU pods have slow cold starts (model load) — mitigate with NIMCache/PVCs
  • Scale-up needs free GPU nodes → pair HPA with cluster autoscaler

🔀Alternative

KEDA — the common alternative for event/queue-driven scaling (async agent workloads on queue depth).

Tensor Parallelism vs Pipeline Parallelism — the exam's favorite axis

TP — TENSOR PARALLELISM

  • Lowers per-request latency (TTFT and ITL); splits weights and KV cache
  • Typical: 2–8 GPUs within one NVLink island
  • Signal words: "minimize TTFT", "NVLink", "2-8 GPUs"

PP — PIPELINE PARALLELISM

  • Raises throughput / enables bigger models, but adds latency (pipeline bubbles = a stage sits idle waiting on the previous one — GPU1 can't start layers 41–80 until GPU0 finishes 1–40 for that token)
  • Typical: across nodes, or PCIe-only boxes
  • Signal words: "two nodes", "PCIe only", "doesn't fit"

📐Rule of thumb

TP = GPUs per node, PP = number of nodes for models too big for one node. Combine TP×PP multi-node — e.g., 405B: TP8 × PP2.

🧩Also exists

Expert parallelism for MoE models; data parallelism = just more replicas behind a load balancer — that's horizontal scaling, not model splitting.

MIG (Multi-Instance GPU) = the opposite: hardware-partition one A100/H100 into up to 7 isolated instances (own compute + VRAM + bandwidth) to pack many small models on one GPU. K8s sees each slice as nvidia.com/mig-1g.10gb etc.; GPU Operator's MIG manager sets profiles. TP/PP spread one big model across GPUs; MIG splits one GPU among many small pods — a TP model can't fit in one slice.

⚖️Load balancing for LLMs

Round-robin is naive (request costs vary wildly with prompt/output lengths) — prefer least-outstanding-requests or KV-cache/prefix-aware routing (NVIDIA Dynamo KV-aware router, gateway-API inference extensions). Sticky/affinity routing improves prefix-cache hits for agents reusing long system prompts. Multi-node TP/PP groups = one logical replica (LeaderWorkerSet pattern).

Throughput vs latency & batch-size tuning

The throughput–latency curve (find the knee)

Method: GenAI-Perf sweep of concurrency → plot curve → pick the knee that meets your SLO. max_batch_size caps in-flight requests; if concurrency > max_batch_size × replicas → requests queue and TTFT explodes.

Agent-specific scaling — why request-count under-counts

📐Variable-length execution

A classifier = one fixed forward pass. An agent plans a multi-step run with a variable number of LLM + tool calls. Simple Q ≈ 1 LLM call (~2 s); research task ≈ 8 LLM + 15 tool calls (~90 s). Two requests can differ 50× in work → request count badly under-estimates load.

Scale on the work signals instead:

  • In-flight (active) agent executions
  • GPU inference queue depth
  • Token throughput (consumed vs available)

Token-budget-aware: 50K-token budget × 100 concurrent ≈ 5M in-flight tokens → size NIM replicas on token throughput, not RPS. Multi-model: scale the 70B reasoner, the classifier, and the embedder independently — don't over-provision the cheap models to match the bottleneck.

🚧Backpressure, rate limiting & probes

  • A single complex request can burn 100K+ tokens → use token-aware rate limiting, not request-count limiting
  • Backpressure: when the queue is full, return HTTP 429 / 503 + Retry-After — don't accept work you can't serve
  • Liveness = "process alive?" (restart) vs Readiness = "can take traffic?" (pull from LB)
  • NIM readiness is special: pod running but not ready for minutes while it loads tens of GB into GPU → generous initialDelaySeconds, gate LB on /v1/health/ready
  • Graceful-drain on shutdown: agent runs carry state (history, tool results) — a mid-run pod kill loses the whole multi-step execution

Section 6 · The two ends of the spectrum

Edge (Jetson + Nemotron Nano) ↔ DGX Cloud (NVCF)

🤖Edge deployment

  • Why: physical-world agents (robots, inspection, kiosks) need low latency, privacy/data locality, offline operation — no datacenter round-trip
  • Platform: Jetson modules (Orin Nano/NX/AGX → Jetson Thor) running JetPack (CUDA/TensorRT on aarch64)
  • Frameworks: TensorRT-LLM / llama.cpp / Ollama on-device; some NIMs supported on Jetson-class devices
  • Nemotron Nano — NVIDIA's small open reasoning family purpose-built for edge/local agents: Nemotron Nano 9B v2 (hybrid Mamba-Transformer), Nemotron 3 Nano 4B. Tool calling + reasoning at small size; runs on Jetson Orin Nano/Thor, DGX Spark, RTX PCs

☁️DGX Cloud & Serverless Inference (NVCF)

  • NVIDIA's managed AI platform: reserved GPU clusters hosted in partner clouds (AWS/Azure/OCI/GCP) with the full NVIDIA AI Enterprise stack — train/fine-tune/serve without owning hardware
  • Key deployment piece: DGX Cloud Serverless Inference, built on NVIDIA Cloud Functions (NVCF)
  • Deploy NIMs, custom containers, or Helm charts behind a single API
  • Auto-scales including scale-to-zero — no charge while idle, no charge for cold-boot time
  • Multi-cloud / multi-cluster — one function can burst across DGX Cloud and partner GPU capacity
  • Choose it for elastic, bursty agent inference without operating Kubernetes yourself
Section 7 · Ship it safely

CI/CD for Agents

An "agent release" = code version + prompt version + model version + engine version — one artifact. Pin image tags + model versions; Triton↔TRT-LLM container versions must match. Model/prompt changes can regress behavior without any code diff → CI runs eval suites (task success, tool-call correctness, safety) before promotion.

📦Containerization

Agent code, tool servers (MCP), and model servers ship as separate images; GPU pods request nvidia.com/gpu; pin tags for reproducibility.

🏷️Model versioning

Triton model repo numeric version dirs + version_policy (latest / all / specific — serving 2 versions simultaneously enables A/B). NIMs = versioned container tags on NGC; registries (NGC, MLflow) track lineage.

Rollout strategies — pick by constraint

Blue-Green INSTANT ROLLBACK

Two identical environments; flip 100% after validation. Instant rollback (flip back) but 2× capacity — expensive with GPUs. Best for big, risky changes.

Canary LOW BLAST RADIUS

Small live slice first, watch metrics plus agent-quality signals, ramp 10→25→50→100%. Lowest blast radius; slower; needs good observability.

Rolling K8S DEFAULT

Gradual pod replacement. Medium risk, no 2× cost. Routine minor stateless updates.

Shadow / Mirror ZERO USER RISK

New version gets a copy of traffic; responses discarded. Zero user exposure, evaluate on real traffic.

Builder's corner — habits that also win exam points

♻️Exploit prefix caching deliberately

Agents resend huge system prompts + tool schemas every step. Paged-KV block reuse (TRT-LLM/vLLM) + cache-aware/sticky routing across replicas can cut TTFT dramatically — keep the static prompt prefix byte-identical across calls.

📏Benchmark with your real traffic shape

GenAI-Perf sweep with your actual ISL/OSL (agents = long inputs, short outputs); pick batch size + replicas from the throughput–latency knee meeting your SLO; set HPA thresholds (e.g., gpu_cache_usage_perc) from those curves — never from defaults.

🧬Model + prompt + engine = one versioned artifact

Pin NIM tags, TRT-LLM↔Triton versions, and prompt versions together; promote through CI only after eval gates (task success, tool-call validity, guardrail violations); canary on quality metrics, not just 5xx rates.

🧊Budget for cold starts & right-size with quantization

GPU pods load tens of GB — pre-cache (NIMCache/PVCs), keep a warm minimum, use scale-to-zero (NVCF) only where cold latency is OK. And try FP8 / INT4 AWQ or a smaller speculative-decoded model before adding TP across more silicon — often halves the GPU bill at near-equal quality.

Section 9 · Don't fall for these

⚠ Exam Traps & Gotchas

⚠ 1 · HPA on CPU/memory for LLM pods

Wrong. GPU inference pods show low CPU while saturated. Scale on gpu_cache_usage_perc, queue depth/concurrency, or TTFT p90 via Prometheus + prometheus-adapter (custom metrics API).

⚠ 2 · GPU Operator vs NIM Operator vs device plugin

GPU Operator provisions GPU infrastructure (drivers, toolkit, device plugin, DCGM); NIM Operator manages NIM microservice lifecycle (caching, deploy, autoscale). Neither serves models itself.

⚠ 3 · Dynamic batching vs in-flight batching

Triton dynamic batching merges requests before execution (stateless models); in-flight/continuous batching swaps requests per token iteration inside the LLM engine (TRT-LLM/vLLM). For LLM generation → in-flight batching.

⚠ 4 · TP vs PP reversed

TP = split within layers, needs NVLink, lowers latency. PP = split by layers/stages, tolerates slow interconnects, raises throughput/capacity but adds latency (bubbles). "Lowest latency on one NVLink node" → TP. "Across two nodes" → TP in-node + PP across.

⚠ 5 · Horizontal scaling vs multi-GPU parallelism

More concurrent users of a model that fits on one GPU → add replicas + load balancer, NOT tensor parallelism. TP/PP is for models too big or too slow on one GPU.

⚠ 6 · Throughput ≠ latency

Bigger batch raises tokens/sec but worsens TTFT/ITL. If the constraint is a latency SLO, "increase batch size" is a trap answer.

⚠ 7 · TTFT vs ITL vs tokens/sec

TTFT = prefill/queue (perceived responsiveness); ITL/TPOT = per-token streaming speed; TPS = system throughput. Long prompts hurt TTFT; long outputs accumulate ITL. Don't accept a tokens/sec answer to a "first response feels slow" problem.

⚠ 8 · Quantization confusions

FP8 needs Hopper (H100)/Ada or newer. AWQ is weight-only 4-bit (activation-aware = uses activation statistics to pick salient weights — it does NOT quantize activations). SmoothQuant = INT8 weights+activations. Quantization mainly buys memory + bandwidth → more KV cache + throughput.

⚠ 9 · Paged KV cache ≠ faster math

It eliminates memory fragmentation/over-allocation, enabling bigger batches and prefix/block reuse — a memory-management technique, not a kernel speedup.

⚠ 10 · Speculative decoding never changes output quality

The target model verifies and accepts/rejects draft tokens, preserving the target distribution. "Trades accuracy for speed" is false. It helps latency-bound/small-batch regimes most, not max-throughput ones.

⚠ 11 · Blue-green vs canary cost/rollback

Blue-green = instant rollback but ~2× resources (expensive with GPUs!). Canary = gradual + cheap but slower. "Instant rollback required" → blue-green. "Limit blast radius, validate on live traffic" → canary. "No user exposure at all" → shadow.

⚠ 12 · Triton ensemble vs BLS

Ensembles are static DAGs; conditional/looping logic (agent-ish control flow) needs BLS (Python backend), not an ensemble.

⚠ 13 · NIM profile auto-selection order

Filter to GPU-runnable profiles first, then rank: backend (trtllm > vllm > sglang) → precision (MXFP4>FP8>…>INT4_AWQ) → latency-optimized default → highest TP. Trap: NIM does NOT default to throughput-optimized. (Current docs put backend before precision; NIM 1.x's older memory→precision→latency→TP→backend chain is superseded — but latency-default and trtllm-preferred hold either way.)

⚠ 14 · Edge sizing

Jetson Orin Nano (8 GB) cannot run a 70B model. The answer is a quantized small model (Nemotron Nano INT4/Q4), or hybrid edge-cloud where the edge handles fast/local skills.

⚠ 15 · NAT vs NIM are not the same thing

NAT = orchestration tier (agent graph, tool routing, memory, guardrails, CPU); NIM = inference tier (model serving, GPU). NAT calls NIM. "Deploy the agent as one container" is wrong — the stack is multi-container so tiers scale independently. Autoscaling the NAT pod on CPU won't help when NIM is the bottleneck.

⚠ 16 · NAT front-end selection

Match the front end to the consumer: HTTP clients → FastAPI; agent-as-a-tool for MCP agents → MCP (or FastMCP lightweight); each agent an independent service → A2A (Agent Cards); dev/debug → Console. Trap: A2A when it's just consumed as a tool (that's MCP), or Console in production.

⚠ 17 · Long agent runs need async jobs, not long timeouts

A 90 s task over a synchronous REST call hits client timeouts, LB disconnects, retry storms. NAT pattern: submit → return a job ID immediately → Dask executes → client polls (or WebSocket). "Synchronous REST with a 120 s timeout" is the trap distractor.

⚠ 18 · Request-count autoscaling under-counts agent load

Agent runs do a variable number of LLM/tool calls — two requests can differ 50×. Scale on token throughput / in-flight executions / queue depth, with token-aware limiting and HTTP 429/503 + Retry-After backpressure — not raw request count. Graceful-drain on shutdown (agent runs carry state).

5HPA scaling on KV-cache utilization (gpu_cache_usage_perc via prometheus-adapter)YAML

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: {name: nim-llm-hpa}
spec:
  scaleTargetRef: {apiVersion: apps/v1, kind: Deployment, name: nim-llm}
  minReplicas: 1
  maxReplicas: 4
  behavior:
    scaleDown: {stabilizationWindowSeconds: 300}   # don't flap on bursty agent traffic
  metrics:
  - type: Pods
    pods:
      metric: {name: gpu_cache_usage_perc}         # NIM's own /v1/metrics gauge, via prometheus-adapter
      target: {type: AverageValue, averageValue: 100m}   # 100m = scale when avg KV-cache usage > 10%

What to notice: the metric type is Pods and a custom metric — not Resource: cpu (trap #1). The 100m is Kubernetes milli-unit notation (the same m as in 250m CPU = ¼ core): 100m = 0.1 of the metric's raw value. Since gpu_cache_usage_perc is reported as a 0–1 fraction, 100m = 0.1 = scale out once average KV-cache occupancy crosses 10% (it does not mean 100% or 100 ms). The full pipeline is NIM /v1/metrics → Prometheus scrape (ServiceMonitor) → prometheus-adapter rule republishing it on the custom-metrics API → HPA; without the adapter rule this YAML applies cleanly but never scales. Pick the averageValue threshold from your GenAI-Perf curves, not from defaults.

7trtllm-build: limits get baked into the engine at compile timeBASH

trtllm-build --checkpoint_dir ./llama3.1-8b-fp8-ckpt \   # produced by quantize.py / Model Optimizer
  --output_dir ./engine-h100-tp1 \
  --max_batch_size 64 --max_input_len 4096 --max_seq_len 8192 \
  --max_num_tokens 8192 \

What to notice: --max_batch_size, --max_input_len, --max_seq_len, --max_num_tokens (and TP/PP, precision, GPU arch from the checkpoint/build host) are frozen into the serialized engine — changing any of them means a rebuild, which is exactly why NIM ships per-GPU "optimized" profiles. The newer PyTorch-backend path (trtllm-serve, LLM() API) skips ahead-of-time engine compilation, trading a little peak perf for flexibility.

8Triton config.pbtxt: dynamic batching + concurrent instances (the two biggest free levers)PBTXT

name: "reranker"
backend: "onnxruntime"
max_batch_size: 64
dynamic_batching {
  preferred_batch_size: [ 8, 16, 32 ]
  max_queue_delay_microseconds: 100   # wait up to 100us to merge requests into a batch
}

  { count: 2, kind: KIND_GPU, gpus: [ 0 ] }   # 2 copies execute concurrently on GPU 0
]

What to notice: this is request-level dynamic batching — right for stateless models (rerankers, embedders, classifiers), wrong answer for LLM generation, where in-flight batching inside TRT-LLM/vLLM is the lever (trap #3). 10Where the LangGraph agent fits: a containerized FastAPI agent that calls the NIM via env varPYTHON

# agent_app.py — CPU pod; the NIM is a separate GPU pod. They scale independently.
import os
from fastapi import FastAPI
from langchain_nvidia_ai_endpoints import ChatNVIDIA   # pip install langchain-nvidia-ai-endpoints
from langchain.agents import create_agent              # LangChain/LangGraph 1.x canonical factory
# (legacy: from langgraph.prebuilt import create_react_agent — deprecated in v1.0, removed in v2.0)

llm = ChatNVIDIA(
    base_url=os.environ["NIM_BASE_URL"],                # e.g. http://nim-llm:8000/v1 (K8s Service DNS)
    model=os.environ.get("NIM_MODEL", "meta/llama-3.1-8b-instruct"),
)
agent = create_agent(llm, tools=[])                     # your tools here; ChatNVIDIA supports bind_tools
app = FastAPI()

@app.post("/invoke")
async def invoke(body: dict):
    result = await agent.ainvoke({"messages": [("user", body["query"])]})
    return {"answer": result["messages"][-1].content}
# Dockerfile: FROM python:3.12-slim; pip install fastapi uvicorn langchain langgraph langchain-nvidia-ai-endpoints
# CMD ["uvicorn", "agent_app:app", "--host", "0.0.0.0", "--port", "8080"]

What to notice: the agent Deployment scales on CPU/RPS with a vanilla HPA while the NIM Deployment scales on gpu_cache_usage_perc — decoupling them via NIM_BASE_URL is the architectural point (one agent request fans out into many LLM calls, so the two tiers have different scaling curves). The 2025–26 consensus is (a) framework wars are settled per-workload, not globally — benchmark your own ISL/OSL (SqueezeBits, Modal, Jack Cook); (b) self-hosting wins decisively for batch/offline tokens and stays hard for interactive serving (Frye, BentoML); (c) on Kubernetes, the action has moved up a layer from "get a GPU pod running" to KV/prefix-aware routing and operator CRDs (llm-d/KServe, KAITO, Anyscale) — exactly the NIM Operator + Dynamo direction NVIDIA is betting on.
