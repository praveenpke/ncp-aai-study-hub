One analogy holds the whole domain together: NVIDIA's stack is a restaurant franchise. Each product has exactly one job in the kitchen.

⚡ CUDA

The electricity and gas lines. GPU programming layer everything compiles against.

🔥 TensorRT(-LLM)

The industrial kitchen equipment tuned for speed — compiles and optimizes inference.

A fully-equipped food truck: engine, recipe, and a standard service window (OpenAI-compatible API) in one box. Visit NVIDIA's food court (hosted on build.nvidia.com) or park the truck in your own lot (self-hosted).

The franchise operations team: Customizer trains the chefs, Evaluator runs taste tests, Guardrails is the bouncer + health inspector, Retriever stocks the pantry, Agent Toolkit is the efficiency consultant with a stopwatch — works with any kitchen brand.

NVIDIA's house recipes in three portion sizes: Nano / Super / Ultra.

⚠ Stack-order trap

TensorRT-LLM is inside NIM; Triton is the serving layer NIM builds on; NIM is inside Blueprints. Any answer placing NIM "below TensorRT" or Triton "above NIM" is wrong. Also: NIM auto-falls-back to vLLM on GPUs without optimized TRT-LLM engines — it still runs, just less optimized.

Section 3

03NIM — The Core Deployment Unit

A pre-built, GPU-optimized Docker container bundling (a) model weights, (b) an inference engine (TensorRT-LLM, vLLM, or SGLang), and (c) an OpenAI-compatible REST API. One model = one NIM container.

Why it exists

Deploying TensorRT-LLM + Triton by hand requires engine compilation, profile tuning, and serving config. NIM collapses all of that into docker run with day-0 support for new models.

docker run --rm --gpus all --shm-size=16GB \
  -e NGC_API_KEY=$NGC_API_KEY \
  -v ~/.cache/nim:/opt/nim/.cache \
  -p 8000:8000 \
  nvcr.io/nim/meta/llama-3.1-8b-instruct:latest

Drop-in OpenAI client swap

Switch from OpenAI to a NIM by changing only base_url and api_key in any OpenAI SDK / LangChain / LlamaIndex client.

from openai import OpenAI
client = OpenAI(
  base_url="https://integrate.api.nvidia.com/v1",
  api_key="nvapi-...")
resp = client.chat.completions.create(
  model="nvidia/llama-3.3-nemotron-super-49b-v1",
  messages=[{"role":"user","content":
    "Plan a 3-step research task."}])

Self-hosting later = change base_url to http://localhost:8000/v1. Nothing else changes.

Section 4

04NeMo Family — The Agent Lifecycle

"NeMo" means two things: the original open-source NeMo Framework (training/alignment library) and the NeMo microservices (GA April 2025) — API-driven services for the enterprise data flywheel: collect → curate → customize → evaluate → guardrail → redeploy, continuously.

change NeMo Customizer — fine-tuning

Microservice (API endpoints on top of NeMo Framework) for post-training: SFT, LoRA (base weights frozen, small trainable adapter matrices injected), P-tuning, and alignment (DPO). Submit a customization job via REST with a dataset (NeMo Data Store) + target model config; output is an adapter/checkpoint you deploy behind a NIM — NIMs can serve multiple LoRA adapters over one base model.

Exam angle: Customizer = training-time improvement. Flagship flywheel use case: distill a large model's behavior into a smaller, cheaper Nemotron Nano using production traffic.

measure NeMo Evaluator

Automated evaluation as a microservice: academic benchmarks (MMLU, BIG-bench, toxicity, multilingual…), custom datasets with standard NLP metrics, and LLM-as-a-judge for open-ended quality. NVIDIA markets "evaluate with ~5 API calls": create target → create config → launch job → poll → fetch results.

Why: closes the flywheel loop — never promote a customized model until Evaluator shows it beats the incumbent. Also evaluates RAG and agent workflows end-to-end (retrieval + answer quality), not just bare models.

NeMo Guardrails — the FIVE rail types (memorize)

Open-source toolkit (pip install nemoguardrails) + microservice for programmable runtime rails around any LLM app. Configured via a folder: config.yml (models + active rails), .co files in Colang (1.0 default; 2.0 newer, python-like dialog modeling), optional actions.py/config.py.

1 · Input rails

Check / sanitize user input: jailbreak detection, PII masking, topic blocking.

2 · Dialog rails

Steer conversation flow via canonical forms (Colang flows).

3 · Retrieval rails

Filter retrieved RAG chunks before they hit the prompt.

4 · Execution rails

Validate tool/action inputs & outputs — the agentic rail.

5 · Output rails

Moderate / fact-check the response before the user sees it.

Built-ins & integrations

Self-check input/output, fact-checking, hallucination detection. Integrates NVIDIA safety NIMs — Llama 3.1 NemoGuard 8B content-safety, topic-control, jailbreak-detect — and 3rd-party (e.g., ActiveFence). NVIDIA's number: up to 1.4x better compliance with ~half a second added latency.

Why Guardrails over competitors

It can model dialog (not just classify single messages) and is LLM-vendor-agnostic. Distinct from fine-tuning safety in via Customizer — that's complementary, not runtime.

NeMo Retriever — the RAG family of NIMs

Embedding NIM

llama-nemotron-embed-1b-v2 — 8,192-token context, multilingual, Matryoshka/dynamic embedding dims. (renamed from llama-3.2-nv-embedqa-1b-v2 in NeMo Retriever NIM 1.13.0; 300M + VL variants also exist).

Reranking NIM

llama-nemotron-rerank-1b-v2 — relevance logits, 3.5x smaller than the older mistral-4b reranker. (renamed from llama-3.2-nv-rerankqa-1b-v2).

Extraction: NV-Ingest

a.k.a. NeMo Retriever Extraction (same thing, two names): scalable pipeline parsing messy enterprise docs (PDF, DOCX, PPTX) into text, tables, charts, images/infographics as structured metadata + chunks. Runs with specialized NIMs (page-elements, table-structure, OCR/PaddleOCR) or a single nemotron-parse model. Engine behind the multimodal PDF blueprints.

NeMo Agent Toolkit — NOT a framework

What it is

Open-source library (pip install nvidia-nat; CLI: nat run, nat serve, nat eval) for connecting, profiling, evaluating, and optimizing teams of agents. It sits alongside LangChain, LangGraph, LlamaIndex, CrewAI, Semantic Kernel, Google ADK, or plain Python — treating agents/tools/workflows as composable function calls described in YAML workflow configs.

Rename chain (know it): AgentIQ → Agent Intelligence Toolkit (AIQ) → NeMo Agent Toolkit. Same tech.

Key capabilities

• Profiler: per-step token usage, latency, cost bottlenecks
• Evaluation harness
• Observability: OpenTelemetry / Phoenix / LangSmith / Weave
• Built-in chat UI, hyperparameter/prompt optimization
• MCP both directions (client + server) and A2A support — the canonical exam answer for "MCP/A2A in the NVIDIA stack."

⚠ The #1 Agent Toolkit trap

It does not replace LangGraph/CrewAI; it wraps and profiles them (framework-agnostic). Any answer implying "rewrite your agents in NeMo Agent Toolkit's framework" is wrong.

05Nemotron — NVIDIA's Open Agentic Models

Open model family (open weights + much of the training data/recipes), post-trained for agentic work: reasoning, tool calling, instruction following. Three sizes, one naming rule. All ship as NIMs on build.nvidia.com and NGC.

TierLlama Nemotron (2025, prior gen)Nemotron 3 (current headline family)Built for
Nano8B (from Llama 3.1 8B)~30B total / ~3B active MoE (31.6B/3.2B), 1M-token contextEdge, RTX PCs, Jetson, cost-sensitive real-time; highest per-$ throughput
Super49B (distilled from Llama 3.3 70B)120B total / 12B active (NVFP4 4-bit on Blackwell; rel. Mar 11, 2026)Best accuracy/throughput on a single (H100) GPU; default for single-agent + multi-agent prod
Ultra253B (from Llama 3.1 405B)~550B total / ~55B active (largest open NVIDIA model)Max accuracy, datacenter/multi-GPU, hardest reasoning/planning/agentic

Reading "total / active"

Nemotron 3 are mixture-of-experts (MoE) models: total params = how much must load into GPU memory; active params = how many run per token (sets latency / throughput / cost). So Nano is 30B total but only ~3B active — it fits and runs cheaply at the edge even though "30B" looks bigger than the old 8B Nano. Size the GPU by total; reason about speed/cost by active.

Signature feature: turn "thinking" on/off per request via the system prompt (detailed thinking on/off) — one model serves both cheap fast paths and deep reasoning paths. Not a separate model download.

Section 8

08AI Workbench vs DGX Cloud

Two tools people confuse. Neither is a model-serving product.

🧰 AI Workbench free · local dev

A free, local-first development environment manager for Windows/macOS/Ubuntu. Each project = a Git repo + container spec; Workbench handles Docker/Podman, CUDA drivers, and JupyterLab/VS Code wiring, and lets you move the same project between laptop RTX GPU and remote/cloud machines.

Example projects: hybrid RAG (toggle between cloud-hosted NIM endpoint and a local NIM), Llama 3 SFT/DPO fine-tuning. Think "dev environment portability" — not a serving platform.

☁ DGX Cloud / Lepton paid · GPU compute

NVIDIA-managed AI training/compute platform hosted on partner clouds (originally renting SuperPOD slices via AWS/Azure/OCI/GCP). Evolved into DGX Cloud Lepton (2025): a GPU marketplace / unified developer platform aggregating compute from NVIDIA Cloud Partners — dev pods, batch training jobs, inference endpoints, multi-cloud, single interface.

Exam-level distinction: Workbench = local dev tool (free); DGX Cloud/Lepton = renting serious GPU capacity as a service.

Section 9

09MCP and A2A in the NVIDIA Stack

MCP = Model Context Protocol (Anthropic-originated open standard connecting an agent to tools/data); A2A = Agent-to-Agent protocol (Google-originated open standard for agents calling other agents across processes/vendors). Rule of thumb: MCP → tools; A2A → other agents. Both are now Linux-Foundation-governed open standards — don't call them "Anthropic's" or "Google's" protocol.

Primary home: NeMo Agent Toolkit

MCP client (call external MCP tool servers from a workflow) and MCP server (expose any toolkit function/agent via MCP, FastMCP-based: nat mcp serve); A2A for cross-process agent teams with authentication.

NIM does NOT speak MCP

NIM speaks the OpenAI API. MCP lives at the agent orchestration layer, not the inference layer.

Blueprints expose MCP too

Newer Blueprints (AI-Q, enterprise RAG) expose retrieval/tools as MCP servers so any MCP-capable agent (Claude, LangGraph, etc.) can call NVIDIA RAG pipelines as tools.

Governance is current-exam material (June 2026)

MCP joined the new Agentic AI Foundation (AAIF) under the Linux Foundation (announced ~Dec 2025), alongside Block's goose and OpenAI's AGENTS.md — vendor-neutral governance, projects keep technical autonomy. A2A was donated by Google to the Linux Foundation and reached v1.0 in early 2026: production-grade Signed Agent Cards (cryptographic domain verification), multi-tenancy (multiple agents per endpoint), and version negotiation (v0.3→v1.0 backward-compatible); 150+ organizations participate.

10NeMo Agent Toolkit — Orchestration Internals

Section 4 says what NAT is ("not a framework"). This is how it's wired — the part the exam drills hardest, because NAT spans Domain 2 (Agent Development) and Domain 7. Everything is a named component referenced by _type in a YAML workflow (or built with nat.builder).

Agent (workflow) types — the named choices

_typePatternPick when
tool_calling_agentUses the LLM's native function-calling API — implicit, fewest tokensModel supports tool calling; you want speed
react_agentExplicit Thought → Action → Observation loop, visible in the traceTransparent reasoning / models without native tool calling
reasoning_agentPlans the whole approach up front, then invokes a function (not between every step)A reasoning model should plan-then-act
rewoo_agentReasoning Without Observation — plans all tool calls before executing anyToken/latency-sensitive multi-tool plans

Full 1.7 orchestrator set also includes sequential_executor, parallel_executor, responses_api_agent, auto_memory_wrapper. (Earlier docs wrote bare tool_calling/react; v1.7 _type strings carry the _agent suffix.)

⚠ Agent-type trap

tool_calling_agent = implicit/native; react_agent = explicit Thought-Action-Observation trace. Swap one YAML line (_type: tool_calling_agent → _type: react_agent) and the trace changes from [llm_call]/[tool_call] to [thought]/[action]/[observation]/[answer]. The trick option labels the native-function-calling agent as "ReAct."

The canonical "first workflow" (CPU-only, hosted NIM)

functions:
  get_current_time:
    _type: python        # docstring = tool description
llms:
  nim_llm:
    _type: nim           # hosted build.nvidia.com or self-hosted
    model_name: meta/llama-3.3-70b-instruct
    temperature: 0.1
workflow:
  _type: tool_calling_agent  # swap to react_agent to see the trace
  llm_name: nim_llm

nat run   --config_file workflow.yaml --input "What time is it?"
nat serve --config_file workflow.yaml   # REST + chat UI (interactive chat)

nat.builder vs YAML — same runtime

Two authoring styles that compile to the identical internal representation. YAML = declarative, diffable, config-as-code, the deployment default. nat.builder (Python) = needed for dynamic behavior YAML can't express (conditional tool registration, runtime-computed middleware) and natural for pytest.

Trap: "YAML behaves differently / runs faster than nat.builder" is false — choosing one changes nothing at runtime.

Field-name caveat: NAT keys shift across releases (tools: vs tool_names:). Memorize the agent-type names and concepts; verify exact keys per version.

Middleware chain — order is a security property

Middleware intercepts every request/response (like HTTP middleware), adding cross-cutting concerns without touching agent logic. Request flows down the chain, response flows back up. Built-ins: logging · defense · caching · red teaming · dynamic dispatch · timeout.

⚠ "Order doesn't matter" is wrong

For a production deployment the canonical order is logging → defense → caching → (red teaming, staging only) → dynamic dispatch. Defense is NAT's bridge to NeMo Guardrails; red teaming runs adversarial probes in staging and is usually off in production.

Plugin system — frameworks as packages

The framework-agnostic claim is architectural: each framework is a separate plugin distribution, not built into core. Install standalone (nvidia-nat-langchain) or via extra: nvidia-nat[adk|agno|crewai|langchain|llama-index|semantic-kernel]; nvidia-nat[all] pulls everything.

Each plugin gives four adapters: LLM wrapper (framework uses NAT's LLM), tool conversion (framework tool → NAT function), callback handlers (events → NAT middleware/observability), parser adapters. A public Plugin API lets third parties add new frameworks without forking (how Agno was added).

Trap: a plugin adapts components, not the whole framework runtime — orchestration/memory/middleware stay NAT's. "A LangChain agent runs identically inside NAT" is wrong.

Authentication — multi-provider

A production workflow authenticates to several services at once. Providers are declared (each named) under authentication:; credentials are loaded at runtime, accessed by name, never logged or persisted. Built-ins:

• API Key — simple service (NIM)
• OAuth2 (Authorization-Code Grant) — enterprise SSO
• Bearer Token — pre-issued service tokens
• HTTP Basic — legacy (TLS only)
• MCP — mcp_oauth2 built-in; needs streamable-HTTP transport (SSE can't carry auth)

Secrets stay out of YAML via _env / ${VAR} refs, filled from Vault/Secrets Manager/K8s at deploy.

nat_test_llm — deterministic testing

LLM output is non-deterministic, so you can't unit-test an agent against a real model. nat_test_llm is a mock LLM returning pre-scripted responses — assert exact tool-call sequences and arguments and test the middleware chain repeatably.

The rule: real LLM for evaluation (quality, via NeMo Evaluator); nat_test_llm for testing (correctness, in CI).

Interactive Workflows & cron

Interactive Workflows run over a bidirectional WebSocket: the agent can pause mid-task for clarification or an approval gate (e.g. "refund \$750 > \$500 — approve?") before a sensitive action. Persist state in the Redis memory backend so a dropped socket doesn't lose context (ping_interval/max_idle).

Cron integration runs a workflow on a schedule (daily reports, periodic data-quality checks) — for non-reactive monitoring/maintenance agents.

Troubleshooting tells (exam-flavored)

🔑 401 Unauthorized

Key unset, expired, or has trailing whitespace/newline. Regenerate at build.nvidia.com.

🔍 404 / "model not found"

Wrong model id or api.nvidia.com instead of integrate.api.nvidia.com. NAT/Guardrails model names need the provider prefix: meta/llama-3.3-70b-instruct.

⏳ 429 Too Many Requests

Free-tier rate limit (~40 req/min class). Back off; don't load-test the catalog.

🐳 NGC unauthorized

Forgot the docker-login username is the literal $oauthtoken, or used the build.nvidia.com key instead of the NGC key.

🛡 Guardrails "engine not recognized"

Version mismatch: engine: nim (newer) vs engine: nvidia_ai_endpoints (older). Or missing colang_version: "2.x", or rail name in config.yml ≠ name in .co.

💥 Self-hosted "CUDA out of memory"

Model too big for the GPU. A 70B needs ~140GB FP16 → two H100s. Use a smaller model or quantized profile.

Builder's corner — field-tested habits

🔁 Prototype hosted, ship self-hosted

Keep base_url/model in env config from day one so the hosted→self-hosted swap is a config change, not a refactor. Watch hosted free-tier rate limits (~40 req/min class) — don't load-test against the catalog.

💾 Cache NIM model stores

📜 Add Guardrails as config, not code

13Exam Traps & Gotchas

Nineteen ways the exam will try to fool you. Read each one twice.

1NIM ≠ a model

A NIM is the container/microservice packaging (engine + weights + API). "Nemotron" is the model; "Nemotron NIM" is its deployable form. "Download the model from build.nvidia.com" usually means pull the NIM container from NGC.

2NeMo Agent Toolkit is NOT an agent framework

It does not replace LangGraph/CrewAI; it wraps and profiles them. "Rewrite your agents in its framework" = wrong. Rename chain: AgentIQ → AIQ → NeMo Agent Toolkit (same tech).

3Customizer vs Evaluator confusion

Customizer changes weights (LoRA/SFT/DPO); Evaluator scores (benchmarks, LLM-as-judge). The flywheel uses both, in that order — but they are separate microservices.

4Guardrails has FIVE rail types

People forget retrieval and execution rails. For agent tool-call safety, the answer is execution rails, not output rails.

5NeMo Retriever is not one model

It's a family: embedding NIM + reranking NIM + extraction (NV-Ingest). NV-Ingest = "NeMo Retriever extraction" — same thing, two names.

6OpenAI-compatible ≠ only OpenAI endpoints; and no MCP

Drop-in swap is base_url + key, but NIM adds NVIDIA-specific extras (/v1/health/ready, /v1/metrics, model profiles). NIM does not expose MCP — MCP/A2A live in the NeMo Agent Toolkit layer.

7Stack-order questions

TensorRT-LLM is inside NIM; Triton is the serving layer NIM builds on; NIM is inside Blueprints. NIM "below TensorRT" or Triton "above NIM" = wrong. NIM auto-falls back to vLLM when no optimized TRT-LLM engine exists.

8Nemotron sizes

Nano=8B, Super=49B (single-GPU sweet spot), Ultra=253B. Trick option swaps Super/Ultra. The reasoning toggle is per-request via system prompt, not a separate model download.

9AI Enterprise branch numbers

Production Branch = every 6 months, 9 months support, monthly patches; LTSB = 3 years, quarterly. "Production Branch supported 3 years" is the trap.

10AI Workbench vs DGX Cloud

Workbench = free local/portable dev environment manager; DGX Cloud (Lepton) = paid GPU compute platform/marketplace. Neither is a model-serving product.

11Hosted endpoints are not production-private

If the scenario mentions regulated/PII data, the hosted build.nvidia.com endpoint is the wrong answer regardless of convenience — self-host the NIM.

12Blueprints are reference code, not managed services

You deploy and own them (Docker/Helm); "NVIDIA operates the virtual assistant for you" is false.

13NAT middleware order is a correctness property

Defense before caching, logging first. "Order doesn't matter if all are present" is wrong: defense-after-caching skips safety on cached hits; logging-after-defense loses blocked-request logs. And cache tool results, never LLM responses.

14NAT agent types

tool_calling_agent = implicit/native; react_agent = explicit Thought→Action→Observation. Don't confuse with reasoning_agent (plans up front) or rewoo_agent (plans all calls before any execution). The trick option calls the native agent "ReAct."

15nat.builder ≡ YAML at runtime

They compile to the same internal representation — authoring style only. "YAML runs faster / behaves differently" is false.

16NAT plugins adapt components, not frameworks

A wrapped LangChain tool does not bring LangChain's runtime/memory/orchestration — those stay NAT's. "A LangChain agent runs identically inside NAT" is wrong.

17nat_test_llm is testing, not evaluation

Mock the LLM for deterministic correctness (assert tool-call sequences). Real LLMs are non-deterministic — reserve them for quality evaluation (NeMo Evaluator), not unit tests.

18Two different keys

NVIDIA_API_KEY (nvapi-, build.nvidia.com) calls hosted endpoints; NGC_API_KEY (ngc.nvidia.com) pulls containers from nvcr.io — login username is literally $oauthtoken. Using the build key to docker pull is wrong.

19Model ids need the provider prefix

meta/llama-3.3-70b-instruct, not llama-3.3-70b-instruct — the bare name yields a 404 in NAT/Guardrails configs.

14Scenario Drills — Test Yourself

Read the scenario, decide your answer, then reveal. These are the exact question shapes the exam uses.

1A fintech must run Llama-class inference fully on-prem on H100s, with CVE-patched containers and someone to call when it breaks.
Self-hosted NIM under an NVIDIA AI Enterprise license (Production Branch). Compliance + support = NIM + NVAIE, not hosted endpoints or DIY vLLM.
2Your LangGraph + CrewAI multi-agent app is slow and expensive; you must find which agent/tool burns the tokens — without rewriting either framework.
NeMo Agent Toolkit profiler. It's framework-agnostic instrumentation/profiling across mixed frameworks.
3Team wants their RAG agent to stop answering off-topic political questions and to verify tool-call arguments before execution.
NeMo Guardrails: dialog/input rails for topic control + execution rails for tool validation. Runtime policing = Guardrails, not fine-tuning.
4You must ingest 2M PDFs full of tables and charts into a vector DB for a multimodal RAG agent.
NV-Ingest (NeMo Retriever extraction) → embedding NIM (llama-nemotron-embed-1b-v2) → Milvus → reranking NIM (llama-nemotron-rerank-1b-v2); start from the multimodal PDF/RAG Blueprint. Extraction at scale is exactly what NV-Ingest exists for.
5A 70B model powers a routing agent; costs are too high. You have months of production logs and want a smaller model with equal task accuracy, promoted only if it proves out.
Data flywheel: NeMo Curator on logs → NeMo Customizer LoRA-tunes Nemotron Nano → NeMo Evaluator gates promotion → deploy as NIM. Classic distillation flywheel question.
6A developer with no GPUs wants to prototype an agent today and later deploy the exact same model in the company VPC unchanged.
Start on build.nvidia.com hosted endpoint (OpenAI-compatible, nvapi key); later pull the same NIM from NGC and just change base_url. The try-hosted-then-self-host path is NIM's core story.
7Order these NAT middleware for production: caching, defense, logging, dynamic dispatch.
logging → defense → caching → dynamic dispatch. Logging first (capture blocked requests too), defense before caching (never serve an unchecked cached response), caching before routing.
8A NAT workflow must call NIM for inference, a corporate SSO for user identity, and a remote MCP tool server.
Three providers under authentication: — API Key (NIM via ${NVIDIA_API_KEY}), OAuth2 Authorization-Code (the SSO), and mcp_oauth2 (the MCP server, over streamable-HTTP). Secrets via _env/${VAR} refs, never inline.
9You need a CI test that deterministically asserts your agent calls order_lookup with {order_id:"12345"} before answering.
Mock the model with nat_test_llm (script the tool-call response) and assert the tool name + args. Real LLMs are non-deterministic — use them for evaluation, not unit tests.
10Your support agent must pause and get a human's approval before issuing refunds over $500, mid-conversation.
A NAT Interactive Workflow over WebSocket with an approval gate; persist state in the Redis memory backend so a dropped connection doesn't lose the conversation.

3NeMo Retriever embed + rerank inside a LangChain retriever

# pip install langchain-nvidia-ai-endpoints langchain-community faiss-cpu
from langchain_nvidia_ai_endpoints import NVIDIAEmbeddings, NVIDIARerank
from langchain_community.vectorstores import FAISS
from langchain.retrievers.contextual_compression import ContextualCompressionRetriever

# Embedding NIM (hosted by default via NVIDIA_API_KEY; pass base_url="http://localhost:8001/v1" for a local NIM)
# Current name (NeMo Retriever NIM 1.13.0+); older blueprints used "nvidia/llama-3.2-nv-embedqa-1b-v2"
emb = NVIDIAEmbeddings(model="nvidia/llama-nemotron-embed-1b-v2", truncate="END")
vs = FAISS.from_texts(chunks, emb)                       # chunks: list[str] from your ingest step

# Reranking NIM compresses a wide candidate set down to the best few
# Current name; older form was "nvidia/llama-3.2-nv-rerankqa-1b-v2"
reranker = NVIDIARerank(model="nvidia/llama-nemotron-rerank-1b-v2", top_n=4)
retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=vs.as_retriever(search_kwargs={"k": 20}),  # over-fetch 20, rerank to 4
)
docs = retriever.invoke("What does the contract say about termination fees?")

What to notice: the canonical NeMo Retriever pipeline — embed wide, rerank narrow — expressed as a standard LangChain ContextualCompressionRetriever, with NVIDIARerank as the compressor. The embed and rerank models here are exactly the ones the exam names (section 3.2); both classes take base_url so the hosted→self-hosted swap rule applies to retrieval NIMs too.

5NeMo Agent Toolkit — register a LangGraph agent, drive it from YAML

# my_pkg/register.py — wrap an EXISTING compiled LangGraph graph as a toolkit function
from nat.builder.builder import Builder
from nat.builder.framework_enum import LLMFrameworkEnum
from nat.builder.function_info import FunctionInfo
from nat.cli.register_workflow import register_function
from nat.data_models.function import FunctionBaseConfig

class MyAgentConfig(FunctionBaseConfig, name="my_langgraph_agent"):
    llm_name: str

@register_function(config_type=MyAgentConfig, framework_wrappers=[LLMFrameworkEnum.LANGCHAIN])
async def my_langgraph_agent(config: MyAgentConfig, builder: Builder):
    llm = await builder.get_llm(config.llm_name, wrapper_type=LLMFrameworkEnum.LANGCHAIN)
    graph = build_graph(llm)                      # your unchanged LangGraph StateGraph().compile()
    async def _run(question: str) -> str:
        out = await graph.ainvoke({"messages": [("user", question)]})
        return out["messages"][-1].content
    yield FunctionInfo.from_fn(_run, description="LangGraph research agent")
# config.yml — the toolkit assembles llm + workflow from names, no code changes
llms:
  nim_llm:
    _type: nim                                    # NIM/build.nvidia.com backend
    model_name: nvidia/llama-3.3-nemotron-super-49b-v1
    temperature: 0.2
workflow:
  _type: my_langgraph_agent                       # the name registered above
  llm_name: nim_llm
pip install "nvidia-nat[langchain]"               # package renamed from aiqtoolkit/agentiq
nat run   --config_file config.yml --input "Summarize the top 3 GPU memory bottlenecks"
nat serve --config_file config.yml --host 0.0.0.0 --port 8001   # REST endpoint + chat UI
nat eval  --config_file config.yml                # add an eval section to score datasets

What to notice: the LangGraph graph is untouched — NAT wraps it as a registered function and the YAML decides which LLM/tools/workflow run, which is exactly the "not another framework" claim made concrete. builder.get_llm(..., wrapper_type=LLMFrameworkEnum.LANGCHAIN) is the bridge that hands your graph a LangChain-flavored client for whatever _type: nim model the YAML names. Older form: before the 1.2 rename the package was agentiq/aiqtoolkit with aiq ... CLI and aiq.* imports — same API shape.

6Nemotron reasoning toggle — one model, two behaviors, via system prompt

def ask(question: str, think: bool):
    return client.chat.completions.create(
        model="nvidia/llama-3.3-nemotron-super-49b-v1",
        messages=[
            {"role": "system", "content": f"detailed thinking {'on' if think else 'off'}"},
            {"role": "user", "content": question}],
        # NVIDIA-recommended sampling: 0.6/0.95 when reasoning is on, greedy when off
        temperature=0.6 if think else 0.0, top_p=0.95 if think else 1.0,
    ).choices[0].message.content

fast = ask("Classify this ticket: 'refund not received'", think=False)  # cheap path
deep = ask("Prove the routing policy can't deadlock", think=True)       # emits <think> trace first

What to notice: the toggle is literally the system prompt string detailed thinking on / detailed thinking off — per request, same deployed NIM, no second model (exam trap #8). With thinking on, the response contains a reasoning trace (wrapped in <think> tags) before the answer, so budget more max_tokens and strip the trace before showing users.

8NAT production surface — middleware chain + multi-provider auth + deterministic test

# pipeline.yaml — concept-level; verify exact keys per NAT version
llms:
  llm:                                # named LLM map; referenced below by name
    _type: nim
    model_name: meta/llama-3.3-70b-instruct
    api_key: ${NVIDIA_API_KEY}        # secret via env ref — never inline
workflow:
  _type: tool_calling_agent           # or react_agent / reasoning_agent / rewoo_agent / router_agent
  llm_name: llm
  max_iterations: 10                  # cap the loop so it can't spin forever
  middleware:                         # ORDER IS A SECURITY PROPERTY
    - type: logging                   # 1. first — even blocked requests get audited
    - type: defense                   # 2. before caching — Guardrails input/output + PII
      config: { input_guardrail: nemo_guardrails, block_pii: true }
    - type: caching                   # 3. cache TOOL results, not LLM responses
      config: { cache_tool_results: true, cache_llm_responses: false, ttl: 3600 }
    - type: dynamic_dispatch          # 4. closest to the agent — route by intent
authentication:                       # several providers, each named, secrets via _env
  nvidia_nim:     { type: api_key,   key_env: NVIDIA_API_KEY }
  enterprise_sso: { type: oauth2,    client_id_env: OAUTH_CLIENT_ID, client_secret_env: OAUTH_CLIENT_SECRET }
  mcp_tools:      { type: mcp_oauth2 }  # streamable-HTTP (SSE can't carry auth)
# Deterministic test — assert the tool-call sequence WITHOUT a real LLM
mock = nat_test_llm(responses=[
    {"tool_calls": [{"name": "order_lookup", "args": {"order_id": "12345"}}]},
    {"text": "Your order #12345 ships March 15th."},
])
result = WorkflowTestHarness(config="pipeline.yaml", llm_override=mock).run("Where is order 12345?")
assert result.tool_calls[0].name == "order_lookup"
assert result.middleware_logs["defense"].blocked is False

What to notice: this is the orchestration layer the exam tests in Domains 2 + 7. The middleware list order is the answer to "place these middleware correctly" — logging → defense → caching → dispatch. authentication: shows three providers coexisting with secrets pulled from env, never hardcoded. nat_test_llm makes the agent deterministic so CI can assert the exact tool name + args — testing (correctness), distinct from NeMo Evaluator (quality). Exact YAML keys (middleware:, dynamic_dispatch) are version-dependent; the concepts and ordering are what's stable.
